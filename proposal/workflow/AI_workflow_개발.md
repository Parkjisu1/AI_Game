# AI Workflow - 코드 생성 (Code Workflow 원본)

> **Purpose**: 자사 프로젝트 소스를 활용한 AI 기반 코드 자동 생성 파이프라인
> **Status**: 원본 보존 문서 — 현재 시스템은 이 원본을 기반으로 확장됨 (→ `AI_workflow_통합_구조.md` 참조)

---

## WORKFLOW 개요

- 자사 프로젝트 소스를 활용하여, AI가 사용할 Base Code DB 데이터로 가공 후 축적
- AI를 통한 기획서 자동 제작, 제작한 기획서를 AI가 사용 가능한 기획 DB 데이터로 가공 후 축적
- Base Code DB와 기획 DB 데이터를 사용한 원시 코드 제작
- 제작된 원시 코드를 사용자가 검수 후 문제 사항이 있을 경우엔 피드백 제공, 제공한 피드백을 기반으로 코드 재생성
- 재생성한 코드 평가
- 평가 정보 기반으로 재생성 코드를 분류하여 DB 데이터로 가공 저장

### 기존 구조에서 수정된 것

기존 구조의 경우, 1개의 AI에게 여러 역할(데이터 가공, 기획서 제작, 코드 생성)을 번갈아 부여하며 사용하는 구조였습니다.
이 경우 역할 전환 과정에서 환각현상이 발생하거나 AI 자체에 에러가 발생할 가능성이 높습니다.
이를 방지하기 위해 AI를 3개로 분리하고, 각각 고정된 역할을 부여하였습니다.

```
1단계 AI: Base Code DB 가공
2단계 AI: 기획서 제작/가공
3단계 AI: 코드 생성
```

각 AI가 하나의 역할에 집중하므로 전문성이 높아지고, 병렬 작업이 가능해집니다.

---

## [1단계] Base Code DB 가공

자사 프로젝트 소스를 활용하여, AI가 사용할 Base Code DB 데이터로 가공 후 축적

### 분류 체계

```
Layer > Genre > 대기능 > 소기능
```

### 방법

자사 프로젝트에 있는 소스들을 Layer로 대분류 한 후, 장르에 대해 따로 소분류, 그리고 분류한 결과물들을 카테고리에 따라 소분화 해서 Tag로 라벨링, AI가 참고해서 사용할 수 있는 자료로 가공 후 축적.

### 과정

가공할 데이터(소스코드)가 들어있는 폴더의 경로를 지정하면, AI가 아래 규격에 맞춰 자동으로 가공하여 저장

### 기존 단계에서 변경한 항목

#### 1) 시스템 → Layer

이유: 자사 프로젝트로 AI를 통한 데이터 증축 과정에서, 다른 장르지만 같은 역할을 하는 것들에 대한 분리가 넓은 범용성, 확정적인 범주의 충돌로 인해 환각 현상이 발생.
그에 따라 범용성과 범주가 충돌이 발생하지 않는 층이라는 개념으로, 수평적 데이터의 분리가 가능한 제일 본질적인 단위로 수정했습니다.

#### 2) Genre 추가

이유: 같은 기능이라도 장르에 따라 구현 방식이 다를 수 있습니다(예: RPG 인벤토리 vs 퍼즐 인벤토리).
장르를 분류 기준에 추가하여 자료 분리의 정밀도를 높였습니다.
또한 Generic(공통) 장르를 두어 장르 무관한 기능은 공통으로 관리하고, 검색 시 "해당 장르 → Generic 순서"로 참조하여 데이터 부족 문제를 방지합니다.

#### 3) 소기능 Tag 카테고리 추가

이유: 자사의 프로젝트(애쉬엔 베일, 뉴포리아, 잔망루피)를 AI에게 학습시키고, 게임의 종류와 상관 없이 모든 소스를 아울러 특정 지을 수 있는 Tag에 대해 요청.
얻은 정보를 기반으로, 소기능에 "생성, 제거, 반복처리, 집계처리"를 추가하게 되었습니다.
그에 따라 소기능 Tag 분류에서 "생성, 제거, 반복 처리, 집계 처리"를 Tag에 추가되므로써, 누락될 수 있는 코드들을 방어할 수 있는 구조가 되었습니다.

#### 변경에 대한 예상 결과

"Layer > Genre > 대기능 > 소기능" Work flow와 추가한 Tag 분류 카테고리로 구조를 설정하므로써, AI가 전체적인 코드 제작과 재사용 및 가공에 있어서 환각현상이 발생하는 것을 방어함과 동시에 BaseCode를 제작하는 과정인 임베딩에서도 데이터의 알고리즘 소분화로 빠른 진행이 가능해집니다.

### Layer 분류

폴더, 파일 키워드와 상속 패턴을 기반으로 구별 (코드라는 개념에서 제일 큰 단위)

의도: 가장 큰 단위로 구분하므로써 범용성을 넓힘

| Layer | 정의 |
|-------|------|
| Core | 싱글톤, 풀링, 이벤트, 유틸, 확장, 베이스 |
| Domain | Battle, Character, Inventory, Quest, Shop, UI, Audio, Network, Save/Load, Skill, Item, Stage, Reward (프로젝트별 추가 항목은 AI가 판단) |

### Role 판단 가이드라인

AI가 분석하는데 사용할 Role 판단 가이드라인 학습 자료 제공, AI가 가이드에 따라 Role 산정.
(가이드라인을 제공하므로써 Role 역할 정규화)

| Role | 정의 |
|------|------|
| Manager | 시스템 총괄, 전체 흐름 제어 |
| Controller | 개별 개체 제어, 입력 처리 |
| Calculator | 수치 연산, 공식 적용 |
| Processor | 데이터 가공, 일괄 처리 |
| Handler | 이벤트 수신 및 처리 |
| Listener | 이벤트 감지 및 반응 |
| Provider | 데이터/리소스 제공 |
| Factory | 객체 생성 담당 |
| Service | 외부 연동, API 통신 |
| Validator | 유효성 검사, 조건 판정 |
| Converter | 데이터 형식 변환 |
| Builder | 복잡한 객체 단계별 생성 |
| Pool | 객체 풀링 관리 |
| State | 상태 정의 및 전환 |
| Command | 실행 가능한 명령 단위 |
| Observer | 변화 감지 및 알림 |
| Helper / Util | 보조 기능, 유틸리티 |
| Wrapper | 외부 라이브러리 래핑 |
| Context | 실행 환경/상태 정보 보관 |
| Config | 설정값 정의 및 관리 |
| UX | 연출 |

이렇게 하면 AI가 임의로 Role을 만드는 게 아니라, 가이드라인 내에서 선택하게 됩니다.

```
예시: BattleManager → [Core: 싱글톤] [Domain: Battle] [Role: 전투 시스템 총괄]
```

### Genre 분류

파일의 키워드 기준. Layer로 선별된 데이터들을 장르로 구별을 하므로써, 장르 변경 상황에서도 사용할 데이터에 대한 환각현상이 발생하는 것을 차단.
데이터 베이스로 사용할 코드들이 자사의 프로젝트들이기 때문에, 해당 장르들 선정

```
Generic(공통) / RPG / Idle / SLG / Simulation / Tycoon / Merge
```

### 대기능 / 소기능 분류

매서드의 이름과 구조를 기준

**대기능** (소기능들을 포함한 한 단계 큰 단위):

| Tag | 정의 |
|-----|------|
| 상태 변환 | StateControl |
| 값 계산 | ValueModification |
| 조건 확인 | ConditionCheck |
| 자원 이동 | ResourceTransfer |
| 데이터 동기화 | DataSync |
| 흐름 제어 | FlowControl |
| 반응 트리거 | ResponseTrigger |

**소기능** (가장 작은 동사 단위로 구분):

| Tag | 정의 |
|-----|------|
| Compare | 비교 |
| Calculate | 연산 |
| Find | 검색 / 조회 |
| Validate | 유효성 검사 |
| Assign | 할당 |
| Notify | 알림/이벤트 생성 |
| Delay | 지연/대기 |
| Spawn | 생성 |
| Despawn | 제거 |
| Iterate | 반복 처리 |
| Aggregate | 집계 처리 |

### 임베딩된 자료 검수

| 항목 | 설명 |
|------|------|
| 중복 확인 | 같은 내역이 작성되어 있는지 |
| 누락 확인 | 있는 기능인데 추가가 안 되어 있는지 |
| 데이터 포멧팅 확인 | Tag 시스템, Layer, Genre, Domain이 정확한 값을 가지고 있는가 확인 |

## [2단계] 기획서 제작/가공

AI를 통한 기획서 자동 제작, 제작한 기획서를 AI가 사용 가능한 기획 DB 데이터로 가공 후 축적

### 의도

기획서라는 자연어 더미를 명료한 입력으로 필터링, 정규화된 데이터로 가공하므로써, AI가 참조해서 사용하는데 있어 환각현상이 발생하는 것을 방지.

### 과정

```
1. 게임 URL/컨셉/레퍼런스 입력
   → AI가 기획서 제작에 있어서 필요한 최소의 정보 제공

2. 게임 전체 기획서
   → 제공한 결과물에 대해 사람이 확인할 수 있고, 결정할 수 있는 기획서를 AI가 제공

3. 시스템 명세서 작성
   → 자연어로 제공한 기획서를 AI가 사용할 수 있는 구조로 가공
   - 시스템 도출: 프로젝트에 어떤 기능들이 있는지 전부 나열
   - Layer 분류: 1단계의 "Layer > Genre > 대기능 > 소기능" 구조와 동일하게 분류
   - 연결 관계: 코드끼리 연결되지 않는 에러를 미연에 방지

4. AI 기획서 작성
   → 정리된 명세서를 조합하여 AI가 코드 생성에 사용할 구조체로 가공

5. AI_기획서 제작 (노드별 YAML)
   → 구조체를 검색 결과가 빠른 YAML 파일로 생성
```

### 기획서 구조를 이렇게 설정한 이유

게임 개발 경험과 AI의 피드백을 통해 코드 제작의 방향성에 영향을 주는 요소들을 추려서, 기능 단위 구간별로 분리한 구조입니다.
다음과 같은 구조는 AI가 기획서, 명세서를 제작하는데 있어, 형식적인 포맷과 정규화된 데이터 구조를 제공하기 때문에, 극단적으로 생각이 확장되어 환각현상이 발생하는 것을 억제할 수 있습니다.

### 기획서 구조

```yaml
공통 섹션:
  - 게임 개요
  - 핵심 경험
  - 핵심 루프
  - 세션 구조
  - 수익 모델

장르별 섹션:
  RPG: [전투, 성장, 캐릭터, 콘텐츠]
  퍼즐: [보드, 매칭, 스테이지, 메타성장]
  캐쥬얼: [조작, 목표/점수, 난이도]
  시뮬레이션: [자원, 건설, 시장 매커니즘]

선택 섹션:
  - 스토리
  - 튜토리얼
  - 멀티플레이
  - 라이브 운영
```

### 명세서 구조

| 섹션 | 내용 |
|------|------|
| 시스템_목록 | Core, Domain, Game 분류 — 전체 시스템 리스트 |
| 시스템_상세: 목적 | 이 시스템이 왜 존재하는가 |
| 시스템_상세: 책임 | 이 시스템이 담당하는 것들 |
| 시스템_상세: 상태 | 관리하는 데이터/변수 |
| 시스템_상세: 행위 | 수행하는 동작 (트리거, 처리, 결과) |
| 시스템_상세: 연결 관계 | uses, usedBy, extends, events |
| 연결_그래프 | 전체 관계 — 시스템 간 연결 시각화 |

### AI_기획서 구조 (노드별 YAML)

| 섹션 | 내용 |
|------|------|
| metadata | nodeId, version, phase — 노드 식별 정보 |
| dependencies | internal, external — 의존성 목록 |
| contract | provides, requires — 제공/필요 API |
| tags | layer, system, functions — 분류 태그 |
| logicFlow | step, tag, action, next — 실행 흐름 |
| referencePatterns | source, pattern — 참조할 기존 코드 |
| codeHints | patterns, avoidPatterns — 코드 작성 힌트 |

## [3단계] 코드 생성

Base Code DB와 기획 DB 데이터를 사용한 원시 코드 제작

### 의도

저장해 놓은 DB들을 기반으로 환각현상으로 인한 방향성 상실 방지와 코드끼리의 유기적인 연결을 구현.
(작업 과정을 5단계로 소분화 하였습니다.)

### 과정

```
1. AI_기획서 파싱
   → 어떤 방향성으로 코드 제작에 들어갈 것인지에 대한 정보를 학습 후 단계별 메타데이터 추출

2. Base Code DB 검색
   → 추출한 메타데이터를 통해 Base Code DB에서 사용할 DB 선정

3. 코드 생성
   → 선정한 데이터를 가지고 AI가 기획 방향성에 맞는 코드 제작

4. 1차 자가 검증
   → 제작한 코드들을 가상 환경에서 스스로 디버깅

5. 체크 포인트 저장
   → 작업 히스토리를 남김으로써, 다음 단계로 넘어갈지 혹은 환각 방지 자료 생성
```

### 방법

참조에 사용할 AI_기획서 파일 경로 지정해서 요청 시 AI가 코드를 자동 생성 (DB 참조 + 규칙 기반 생성)

### DB 검색 우선순위

Claude 공식 포럼에 있는 "검색" 알고리즘 원리를 이용하여 구상한 순위표입니다.
해당 순위표를 이용하여 AI가 질 높은 정보를 사용해서 코드를 제작할 수 있게 흐름을 설계하였습니다.

```
Expert DB: 6단계 혹은 4단계에서 통과한 코드들만 구조체로 가공 후 저장한 저장소
Base DB:   1단계에서 진행한 결과를 저장한 저장소
```

| 순위 | 검색 대상 | 조건 |
|------|-----------|------|
| 1 | Expert DB (해당 장르) | genre 일치 AND score >= 0.6 |
| 2 | Expert DB (Generic) | genre = Generic AND score >= 0.6 |
| 3 | Genre Base DB | genre 일치 |
| 4 | Generic Base DB | genre = Generic |
| 5 | AI_기획서 기반 생성 | 참조 코드 없음 → logicFlow 의존 |

## [4단계] 피드백/재생성

제작된 원시 코드를 사용자가 검수 후 문제 사항이 있을 경우엔 피드백 제공, 제공한 피드백을 기반으로 코드 재생성

### 의도

Expert DB에 최상위 상태 코드 축적

### 과정

```
1. 코드 리뷰
   → 제작한 코드를 살펴보면서 최적화, 에러 가능성 등의 구체적인 정보 정리
   → 피드백이 있을 경우 피드백 작성 진행
   → 피드백이 없을 경우 [6단계] 진행

2. 피드백 작성
   → 피드백 형식에 맞춰 정리한 사항 작성

3. 코드 재생성
   → 피드백 자료를 기반으로 소스에서 피드백 관련 부분만 재생성
   (피드백 DB가 충분히 축적된 후에는 자연어 피드백 → AI가 정해진 포맷으로 자동 변환 예정)
```

### 피드백 형식

실제 개발 경험에서 느낀 AI가 코드 재생성에 있어 필요로 하는 핵심 피드백 부분을 정리하여 작성한 형식입니다.

| 항목 | 설명 | 예시 |
|------|------|------|
| nodeId | 피드백 대상 노드 | BoardManager |
| genre | 노드의 장르 | RPG |
| category | 에러 유형 | LOGIC.NULL_REF |
| line | 에러 라인 (선택) | 85 |
| description | 에러 설명 | pieces 배열 null 체크 누락 |
| contractChanged | Contract 변경 여부 | false |

## [5단계] 재생성 평가

재생성한 코드 평가

### 의도

피드백을 통해 재생성한 코드 평가를 통해, 전 단계의 피드백 사항이 문제가 없는지, 문제 없는 피드백을 넣었는데도 결과가 이상하면 어디가 잘못된 건지에 대한 파이프라인 평가와 Expert DB에 최상위 상태 코드를 축적.

### 과정

```
1. 재생성 코드 실행
   → AI가 가상 환경에서 재생성한 코드의 에러 유무 및 피드백 사항 적용됐는지 확인 후 주석 추가
   → 주석은 6단계에서 해당 코드를 평가할 기준 (리뷰 주석이 많을수록 신뢰도가 낮음)
   → 에러/피드백 잘 적용된 경우 → [6단계] 진행
   → 에러/이슈가 있는 경우 → 2번 진행

2. 이슈 목록 정리
   → 재생성한 코드에서 발생한 이슈들과 피드백 사항들을 순차적으로 비교하면서 목록 정리
   → 지속적으로 발생하는 이슈나 잠재적인 이슈에 대한 예외 처리가 가능

3. 코드 재생성
   → 정리된 사항들(피드백 기반)을 기반으로 코드 재생성
   → 이전 버전과의 차이점 작성
```

## [6단계] DB 축적

평가 정보 기반으로 재생성 코드를 분류하여 DB 데이터로 가공 저장

### 의도

검증 완료된 코드를 재사용 가능한 구조체로 가공하여 Expert DB에 축적하므로써, AI들이 참조할 퀄리티 높은 DB 구축

### 과정

```
1. 신뢰도 점수 계산
   → 제작한 코드를 "신뢰도 점수 체계"에 따라 점수를 측정

2. 코드 분기
   → 특정 점수를 기준점으로 (Anthropic 자사에서 사용하는 output 평가 점수가 0.4)
   → 해당 점수에 도달하지 못했을 경우 → 저장하지 않음
   → 해당 점수에 도달했을 경우 → [1단계]와 같은 방법으로 Expert DB에 저장

3. Rules 추출
   → 전체 플로우를 구동하면서 쌓인 피드백을 정리해서 반복되는 패턴을 조건부 피드백 사항으로 저장
   → AI 학습 및 방향성 제시 시 개발자가 참고하기 위한 자료
```

### 신뢰도 점수 체계

Anthropic에서 초기 AI를 학습시켰을 때 사용하던 보상체계를 그대로 인용해서 가져왔습니다.

| 이벤트 | 점수 변동 |
|--------|-----------|
| 초기 저장 | 0.4 |
| 피드백 반영 완료 | +0.2 |
| 재사용 성공 (1회당) | +0.1 |
| 재사용 실패 (1회당) | -0.15 |
| 다른 장르에서 재사용 성공 | +0.1 (Generic 승격 검토) |

### Rules 추출 자료

| 규칙 유형 | 설명 | 예시 |
|----------|------|------|
| Generic 규칙 | 장르 무관 공통 규칙 | null 체크 패턴 |
| 장르별 규칙 | 특정 장르에만 적용 | RPG 버프 스택 처리 |
