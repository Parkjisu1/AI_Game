> **검증 기준**: 문서 자체의 논리 완결성, 실행 가능성, 후속 단계(1~8단계)와의 연결성

## 총평

전체적으로 0단계의 **설계 의도(Why)**와 **저장 구조(What)**는 잘 정의되어 있습니다. 코드 Workflow에 없는 기획 고유의 인프라 단계로서 존재 가치가 명확합니다.

아래는 문서 자체의 논리 완결성과 실행 관점에서 **잘 된 부분**, **갭(미정의/미흡)**, **고유 강점** 순으로 정리한 결과입니다.

**이슈 총괄: CRITICAL 1건 / HIGH 3건 / MEDIUM 5건 / LOW 4건 = 13건**

### 1-1. 0단계 필요성 (L218~225)

**v1.2 L221~224:**
```
같은 Workflow 2단계(밸런스 기획)를 타더라도:
- 퍼즐팀: "레벨별 클리어율, 빈 용기 수, 분류 대상 종류 수"가 핵심 파라미터
- Idle RPG팀: "레벨별 HP, ATK, 경험치 곡선, 데미지 공식"이 핵심 파라미터
```

0단계 존재 이유가 한 문단으로 납득됩니다. 같은 문서의 1~8단계가 "How(프로세스)"만 정의하고, "What(장르별 내용물)"은 정의하지 않고 있었으므로 이 갭을 정확히 짚었습니다.

### 1-2. How/What 계층 (L227~233)

**v1.2 L228~232:**
```
Workflow (How) ─── 모든 팀 공통, 프로세스 정의
    │
    └── 장르별 설계 표준 (What) ─── 장르 단위, DB 스키마 정의
            │
            └── 디렉션 히스토리 ─── 사람 단위, 자동 축적
```

3계층 구조가 깔끔합니다. Workflow(팀 공통) → 설계 표준(장르 단위) → 디렉션(개인 단위)로 범위가 좁아지는 설계입니다.

### 1-3. 산출물 4단계 (L243~265)

**v1.2 L246~264:**
```
장르별 설계 표준 문서
├── 구성요소 맵: "이 장르의 게임은 어떤 요소들로 구성되는가"
├── 파라미터 정의: "각 구성요소를 어떤 변수로 조정하는가"
├── 공통/커스텀 구분: "타이틀 간 이식할 것 vs 매번 새로 만들 것"
└── DB 스키마: "위 파라미터들이 Base Design DB에 어떤 형식으로 저장되는가"
```

구성요소 맵 → 파라미터 → 분류 → 스키마 순서가 논리적으로 쌓입니다. 각 단계의 산출물이 다음 단계의 입력이 되는 구조입니다.

### 1-4. AI Tester 역할 분담 (L293~305)

**v1.2 L295~304:**

| 작업 | AI Tester | 사람 |
|------|:---------:|:----:|
| 개별 게임에서 구성요소 관찰/추출 | ● | |
| 타이틀 간 공통/차이점 비교 | ● | |
| 관찰 결과를 장르 체계로 추상화 | | ● |
| 설계 의도 파라미터 정의 | | ● |

AI(관찰/추출/비교)와 사람(추상화/선별/정의)의 경계가 명확합니다.

### 1-5. 디렉션-결과 쌍 (L353~355)

**v1.2 L353~355:**
```
디렉션과 결과(신뢰도 점수, 피드백 횟수, 사후 라벨)가 자동 연결되므로,
시간이 지나면 "어떤 디렉션 방식이 더 나은 결과를 내는가"가 데이터로 드러납니다.
```

"어떤 디렉션이 좋은 결과를 내는가"를 데이터로 측정하겠다는 아이디어가 강력합니다. 코드 Workflow에는 없는 기획 고유 가치입니다.

### 1-6. 코드 Workflow 대칭 (L372~381)

**v1.2 L374~377:**
```
코드: 개발자 피드백 축적 → 반복 패턴 → Rules (조건부 피드백)
기획: 디렉터 디렉션 축적 → 반복 패턴 → Rules (디렉션 패턴)
```

코드의 "피드백 → Rules" 구조와 정확히 대칭됩니다. 같은 문서의 6단계 Rules 추출(L812~820)과도 일관성이 유지됩니다.

## 2. 갭 — 미정의 또는 미흡한 항목

### 2-1. [CRITICAL] 자동 축적 캡처 메커니즘 없음

**v1.2 L367:**
```
자동 축적: 디렉터가 별도로 정리할 필요 없음. AI에게 디렉션을 주는 행위 자체가 기록
```

**문제**: 어떤 시스템이 이것을 캡처하는지 전혀 정의되어 있지 않습니다:
- 디렉션은 어디서 발생하는가? (채팅? 문서 편집? Claude Code 대화?)
- 누가/무엇이 "이것은 디렉션이다"를 판별하는가?
- YAML 구조(L332~343)로 자동 변환되는 과정은?
- target 필드(`Balance.Growth.LevelTable`)는 어떻게 자동 추출하는가?

저장 구조(What to store)는 잘 정의했지만(L330~343), 캡처 프로세스(How to capture)가 완전히 빠져 있어서 **구현이 불가능**합니다.

**보완 방향**: 캡처 메커니즘을 3단계(MVP→반자동→자동)로 정의:
```
L1 (MVP): 디렉터가 "이 디렉션 저장" 명시 → YAML 자동 생성
L2 (반자동): AI가 디렉션 후보 자동 추출 → 디렉터 확인 프롬프트
L3 (자동): 디렉션/질문/잡담 자동 분류 → 디렉션만 저장
```

### 2-4. [HIGH] 디렉션 vs 비디렉션 구분 기준 없음

**v1.2 L367:**
```
AI에게 디렉션을 주는 행위 자체가 기록
```

**문제**: AI에게 주는 발화가 모두 디렉션은 아닙니다:

| 유형 | 예시 | 저장 여부 |
|------|------|-----------|
| 디렉션 | "데미지 공식은 감산식으로, 방어 비중 낮게" | O |

분류 기준 없이 전부 저장하면 노이즈가 쌓여 패턴 추출 품질이 떨어집니다.

**보완 방향**: "생성 결과물의 방향/수치/제약을 지정하는 발화"를 디렉션으로 정의하는 판별 기준 테이블 추가.

### 2-6. [MEDIUM] 필수/선택 구분의 결정권자

**보완 방향**: "합의 불가 시 장르 리드 디렉터가 최종 결정" 1줄 추가.

**문제**: 같은 문서의 물리적 저장 구조(L175~207)에서 `db/` 루트 아래에 `design_standard/`, `design_direction/`을 두고 있는데, 이는 "방향"(L178) 수준으로 확정이 아닙니다. 또한 인덱싱/검색 방안이 없어서, 프로젝트가 많아지면 디렉션 수천 건에서 패턴을 어떻게 찾는지 불명확합니다.

**보완 방향**: 저장 경로 확정 + 인덱스 파일 구조 추가.

## 3. 고유 강점 — 1~8단계에 없는 새로운 가치

### 3-1. 장르별 What 분리 (L218~236)

**v1.2 L235~236:**
```
Workflow 없이 설계 표준만 있으면: 좋은 파라미터 목록은 있는데 활용 프로세스가 없음
설계 표준 없이 Workflow만 있으면: 프로세스는 있는데 AI에게 지시할 구체적 규격이 없음
```

1~8단계는 "어떤 프로세스로 일하는가(How)"를 정의하지만, "장르마다 무엇을 정의해야 하는가(What)"는 각 단계에서 암묵적으로 처리합니다. 0단계는 이 암묵적 부분을 명시적 인프라로 분리합니다.

### 3-2. 디렉션-결과 쌍 (L353~355)

코드 Workflow에서는 피드백이 "이슈 → 수정"의 단방향이지만, 기획 디렉션 히스토리는 "디렉션 → 결과 점수"가 쌍으로 기록됩니다. 시간이 지나면 **어떤 디렉션 방식이 더 나은 결과를 내는지** 데이터로 드러나는 점이 코드 Workflow에도 역으로 도입을 검토할 만합니다.

### 3-3. AI 디렉션 해석 정확도 향상 (L357~359)

**v1.2 L358~359:**
```
같은 디렉터의 과거 디렉션을 참조하면, "이 사람이 '초반은 쉽게'라고 하면
구체적으로 어느 정도를 의미하는지" AI가 학습하여 디렉션 해석이 정확해집니다.
```

디렉터 개인의 언어 패턴을 AI가 학습하는 것으로, DB가 축적될수록 디렉션 → 결과물 변환의 정확도가 점진적으로 향상되는 선순환 구조를 만듭니다.

### 판단

CRITICAL(2-1)과 HIGH(2-2~2-4) 4건은 정본 반영 시 반드시 보완이 필요합니다. 특히 2-1(캡처 메커니즘)이 없으면 0-2 디렉션 히스토리 자체가 구현 불가능합니다.

전체적으로 0단계의 설계 의도와 구조적 가치는 높으므로, 위 13건을 보완하여 정본에 정식 추가하는 것을 권장합니다.
