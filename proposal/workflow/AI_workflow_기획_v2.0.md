## WORKFLOW 개요

자사 및 외부 게임의 기획 자료를 AI가 활용 가능한 Base Design DB로 가공 후 축적
디렉터(기획자)가 상위 디렉션을 제공하면, AI가 Base Design DB를 참조하여 기획서/밸런스/콘텐츠를 생성
생성된 기획 결과물을 디렉터가 검수 후 피드백 제공, AI가 피드백 기반으로 재생성
검증 완료된 기획 자료를 평가하여 Expert Design DB에 축적
AI Tester를 통한 플레이 기반 검증으로 기획 품질 실증
라이브 운영 중 변경된 기획/밸런스 데이터를 DB에 버전 관리하여 반영

### AI 분리 구조

코드 Workflow와 동일한 원칙으로, 역할 전환에 의한 환각현상 방지를 위해 AI를 분리합니다.

```
1단계 AI: Base Design DB 가공 (기획 자료 → 정규화된 DB)
2단계 AI: 기획서/밸런스/콘텐츠 생성 (디렉션 + DB 참조 → 기획 결과물)
3단계 AI: 기획 검증 및 시뮬레이션 (결과물 → 일관성/밸런스/경제 검증)
```

각 AI가 하나의 역할에 집중하므로 전문성이 높아지고, 병렬 작업이 가능해집니다.

### 기획자 역할 재정의: 디렉터 포지셔닝

```
모든 단계의 공통 패턴:

  디렉터(사람): 상위 디렉션 제공 (의도, 방향성, 제약조건)
      → AI: 실행 (DB 참조 → 구체화, 수치 채우기, 양산)
          → 디렉터(사람): 검증 (의도대로인가 판단)
              → 통과: 다음 단계로
              → 피드백: AI 재실행
```

- 상위 시스템 기획은 창의성 + 고민이 필요 → 사람이 담당
- 각 시스템의 기능 구현, 밸런스 수치 채우기, 콘텐츠 양산 → AI가 담당
- 사람은 최종 검증(디렉터 역할)까지 수행

DB가 충분히 쌓이면 디렉션의 상세도가 줄어듭니다:
- DB 부족: "데미지 공식은 ATK×배율-DEF×0.5로, 최소 데미지는 ATK의 5%로 해줘" (수치 레벨까지 지시)
- DB 보통: "데미지 공식은 감산식으로, 레퍼런스는 ㅇㅇ게임 기반인데 방어 비중을 좀 낮춰줘"
- DB 풍부: "RPG 전투 밸런스 잡아줘, 초반은 레퍼런스보다 쉽게" (한 줄 디렉션으로 충분)

### Workflow ↔ 설계 표준 ↔ 디렉션 계층 구조

```
Workflow (How)           — 0~8단계 파이프라인 (이 문서의 실행 절차)
    ↓ 참조
장르별 설계 표준 (What)  — 장르마다 "무엇을 정의해야 하는지" 기준
    ↓ 축적
디렉션 히스토리 (개인)   — 디렉터의 반복 판단 패턴 자동 축적
```

## [0단계] 장르별 설계 표준 + 디렉션 히스토리

### 의도

같은 1~8단계 파이프라인을 타더라도, 장르마다 핵심 파라미터가 다릅니다.
퍼즐팀과 Idle RPG팀이 같은 2단계를 실행해도, "무엇을 정의해야 하는지"의 기준이 다릅니다.
0단계는 이 **"장르별 What"**을 파이프라인 실행 전에 정의하는 인프라 단계입니다.

### 실행 시점

장르 최초 프로젝트 전 1회 실행 + 이후 상시 보완 (8단계 라이브 피드백, AI Tester 발견 시).
매 프로젝트 시작 시 설계 표준을 리뷰하고 필요 시 보완합니다.

### 0-1. 장르별 설계 표준

#### 산출물

```
장르별 설계 표준
├── 구성요소 맵: "이 장르에 필요한 시스템/기능 목록"
│   (예: Idle RPG → 전투, 성장, 장비, 가챠, 경제, 콘텐츠, BM)
│   (예: Puzzle → 코어 메커닉, 레벨디자인, 난이도곡선, 아웃게임, BM, 메타)
├── 파라미터 정의: 각 구성요소의 핵심 설계 파라미터
│   (예: 전투 → {DPS 공식, 공격 속도, 크리티컬, 방어율, ...})
│   (예: BM > IAP > 피기뱅크 → {가격, 해금 조건, 순환 주기, 상품 효율, 보상 구성, 노출 조건, 팝업 우선순위, dynamic pricing 여부})
├── 데이터 스키마 정의: 콘텐츠 양산에 필요한 데이터 구조 정의 (프로그래머 싱크 필수)
│   (예: Puzzle > 레벨디자인 → {레벨 JSON 포맷, 좌표 체계, 오브젝트 타입 enum, 컬러 enum, 보드 조건})
├── 공통/커스텀 구분: 장르 간 공유 가능한 파라미터와 장르 고유 파라미터 분리
├── 도메인별 우선순위: 장르 내 도메인의 기획 우선순위 (Tier 1/2/3)
│   (예: Puzzle → Tier 1: InGame, Content, Balance / Tier 2: UX, BM / Tier 3: OutGame, LiveOps, Social, Meta)
│   (예: Idle RPG → Tier 1: InGame, Balance, OutGame / Tier 2: Content, BM / Tier 3: UX, Social, Meta, LiveOps)
└── DB 스키마: 1단계 Base Design DB의 스키마가 됨
```

> **구성요소 맵 ↔ 기획 도메인 관계**: 구성요소 맵의 항목은 기획 도메인 9종(InGame, OutGame, Balance, Content, BM, LiveOps, UX, Social, Meta)에 매핑됩니다. 도메인 자체는 고정이며, 구성요소 맵은 장르별로 가변입니다. 예: "전투"→InGame, "가챠"→OutGame, "경제"→Balance.

#### 파라미터 상세도 기준

파라미터 정의는 "키워드 나열" 수준이 아니라, **AI가 채워야 할 구체적 항목과 제약조건** 수준이어야 합니다.

```
✗ 부적절한 상세도: "과금 구조, 패키지, 시즌패스, 광고" (키워드 나열)
✓ 적절한 상세도:

BM > IAP > 피기뱅크:
  필수 파라미터:
    - price: 판매 가격 (예: $2.99)
    - unlock_condition: 해금 조건 (예: 20레벨+)
    - fill_cycle: 가득 차는 평균 플레이 횟수 (예: 7~8판)
    - value_ratio: 판매 가격 대비 보상 가치 배율 (예: 2배, 최소 200%)
    - reward_composition: 보상 구성 (예: 100% 유료 재화)
    - exposure_condition: 노출 조건 (예: 가득 찼을 때 팝업)
    - popup_priority: 레벨 클리어 후 팝업 노출 순위 (예: 6순위)
    - dynamic_pricing: 유저별 가격 차등 여부 (예: False)
  선택 파라미터:
    - visual_feedback: 누적 단계별 시각 효과
    - sound_feedback: 청각 피드백
```

이 상세도가 2단계 기획 생성 시 AI에게 "이 파라미터들을 채워라"는 명세가 됩니다.
디렉터는 값의 방향만 지시하고("첫 결제 유도 목적, 저가"), AI가 DB 참조로 구체 수치를 채웁니다.

#### 데이터 스키마 정의

콘텐츠 양산이 필요한 시스템(레벨디자인, 몬스터, 스테이지 등)은 **데이터 구조 자체**가 설계 표준의 일부입니다.
이 스키마는 프로그래머와의 싱크가 필수이며, AI가 콘텐츠를 생성하려면 스키마를 먼저 학습해야 합니다.

```
정의 대상:
  1. 데이터 포맷: JSON/YAML 구조, 필드명, 타입
  2. Enum 정의: 오브젝트 타입, 컬러, 상태값 등의 열거형
  3. 좌표 체계: 좌표 원점, 그리드 구조, 인덱싱 규칙
  4. 제약 조건: 값 범위, 필수/선택 필드, 상호 의존 관계

예시 (퍼즐 레벨 데이터):
  포맷: JSON
  좌표: (0,0)=좌상단, width=y%2==0일 때 11칸/y%2==1일 때 10칸
  버블 타입: Normal=0, Metal=1, Ice=2, ... (30종+)
  버블 컬러: Red=0, Green=1, Blue=2, ... (8종, RandomColor=99)
  보드 조건: 색상 배열, 시작 색상, 별 점수 기준, 무브 수
  빈 칸 처리: 데이터 없음 (None으로 채우지 않음)

프로그래머 싱크 항목:
  - 기존 데이터 포맷 확인 (레거시 호환)
  - 에디터 전용 필드 vs 런타임 필드 구분
  - 값 범위 및 유효성 검증 규칙
```

데이터 스키마가 설계 표준에 정의되면:
- 1단계에서 기존 레벨 데이터를 파싱할 때 스키마 참조
- 2-4단계에서 AI가 신규 레벨을 양산할 때 스키마 준수
- 3단계에서 Validator가 생성된 데이터의 스키마 적합성 검증

#### AI Tester 역할 분담

```
AI Tester (관찰 전문):
  - 외부 레퍼런스 게임에서 파라미터 추출
  - 장르 내 공통 파라미터 패턴 비교
  - 수치 범위 통계 (예: "Idle RPG 10종의 방치보상 최대 축적 시간 = 8~24시간")

디렉터 (추상화 전문):
  - AI 관찰 결과에서 "의미 있는 파라미터" 선별
  - 필수 vs 선택 구분 결정
  - 장르 철학 반영 (예: "이 장르의 핵심 경험은 X이므로 Y 파라미터가 필수")
```

#### 작업 순서

```
1. AI Tester로 핵심 레퍼런스 게임 분석 (파라미터 추출)
2. 추출된 파라미터를 장르 공통/고유로 분류
3. 디렉터가 필수/선택 구분 + 기본값 범위 설정
   → 각 파라미터는 "AI가 채워야 할 항목 + 제약조건" 수준으로 상세화
4. 데이터 스키마 싱크 (프로그래머 협업)
   → 콘텐츠 양산 대상 시스템의 데이터 포맷, enum, 좌표 체계, 제약 조건 정의
   → 기존 프로젝트 데이터 구조 확인 + 에디터/런타임 필드 구분
5. 팀 간 의견 차이: 필수/선택으로 구분하여 수용
   → 합의 불가 시 장르 리드 디렉터가 최종 결정
6. DB 스키마 확정 → 1단계 파이프라인 입력 구조로 사용
```

#### 부트스트랩 경로 (첫 장르)

```
아직 AI Tester DB도 없고 레퍼런스 분석도 없는 상태에서의 경로:

부트스트랩 (첫 장르):
  1. 기획자 경험 기반으로 초안 작성 (AI Tester 없이)
  2. 1~2개 레퍼런스 분석 후 보완
  3. 첫 프로젝트 진행하면서 반복 보완

이후 장르:
  기존 작업 순서(1~5) 정상 실행
```

#### 저장 경로 및 1단계 연결

```

1단계 Step 2(정규화)에서 해당 장르 설계 표준을 참조하여 분류
```

설계 표준 변경 시 semver 적용합니다. 기존 DB 데이터는 소급 적용하지 않으며, 신규 입력분부터 새 표준을 적용합니다.

### 0-2. 디렉션 히스토리

#### 의도

디렉터의 반복 판단 패턴을 데이터로 축적합니다.
**별도 정리 작업 없이, 평소 AI에게 디렉션을 주는 행위 자체가 기록됩니다.**
코드 Workflow의 "개발자 피드백 → Rules DB"와 대칭되는 구조입니다.

#### 축적 대상

```
디렉터가 AI에게 주는 디렉션 발화:
  "초반은 레퍼런스보다 쉽게 해줘"
  "가챠 천장은 반드시 있어야 해"
  "이 BM 구조는 Pay-to-Win이 될 수 있으니 조정해"
```

#### 디렉션 판별 기준

| 유형 | 정의 | 예시 | 저장 여부 |
|------|------|------|-----------|
| 디렉션 | 생성 결과물의 방향/수치/제약을 지정하는 발화 | "데미지 공식은 감산식으로, 방어 비중 낮게" | O |
| 질문 | 정보를 요청하는 발화 | "이전에 비슷한 게임 DB에 뭐가 있어?" | X |
| 확인 | 현재 상태를 확인하는 발화 | "이거 좀 이상한데, 다시 보여줘" | X |
| 잡담 | 업무 외 발화 | "오늘 날씨 좋다" | X |
| 오류 지적 | 버그나 오류를 지적하는 발화 | "여기 오타 있어" | X |

```yaml
  timestamp: "2026-02-27T10:30:00"
```

> **result_score**: 3단계 자동 점수 3종(논리 완결성, 밸런스 안정성, 구현 복잡도) 평균으로 산출됩니다. 7단계 플레이 검증 후 실증 데이터 기반으로 업데이트 가능합니다. 하나의 디렉션이 여러 결과물에 영향을 줄 때는 직접 대상(target)의 점수를 사용합니다.

#### 캡처 메커니즘 (3단계)

```
L1 (MVP): 수동 트리거
  - AI 대화 후 디렉터가 "이 디렉션 저장" 명시 → YAML 자동 생성
  - 최소 부담: 저장할지 말지만 결정

L2 (반자동): AI 추출 + 사람 확인
  - 2~4단계 대화 중 "~으로 해줘", "~방향으로" 패턴 감지
  - "이 디렉션을 기록할까요?" 확인 프롬프트
  - target 필드 자동 추론 (대화 맥락에서)

L3 (자동): 전량 캡처 + 노이즈 필터
  - 디렉션/질문/잡담 자동 분류 (판별 기준 테이블 적용)
  - 디렉션만 저장, 정기적으로 디렉터 확인
```

현재 MVP는 L1(수동) 수준이며, L2 → L3 순서로 자동화를 확장합니다.

#### 축적 가치

```
디렉션 패턴이 축적되면:
  개인: "나는 초반 밸런스를 항상 이렇게 잡더라" → 반복 디렉션 생략 가능
  팀:   "우리 팀은 가챠 천장을 이 범위로 잡더라" → 설계 표준에 반영 가능
  AI:   "이 디렉터가 이런 디렉션을 주면 결과가 좋았다" → 추천 가능
```

#### 패턴 추출

동일 target에 디렉션 10건 이상 축적 시 자동 분석을 제안합니다.

```
분석 절차:
  1. 동일 target의 디렉션을 시간순으로 나열
  2. 반복되는 방향성 패턴 추출
  3. result_score와 교차하여 "좋은 결과의 디렉션 패턴" 식별
  4. 디렉터에게 패턴 확인 요청
  5. 승인 시 Rules DB의 direction_rules 섹션에 저장
  6. 동시에 설계 표준에도 권장 패턴으로 반영 가능
```

#### 코드 Workflow 대응

```
기획: 디렉터의 디렉션 반복 패턴 → 디렉션 히스토리 → 설계 표준 강화
코드: 개발자의 피드백 반복 패턴 → Rules DB → 코드 생성 규칙 강화
```

#### 저장 경로

```
db/design/directions/{director_id}/
├── index.json            # 디렉션 인덱스 (검색용, 경량)
└── entries/
    └── {direction_id}.yaml  # 개별 디렉션 상세
```

#### 개인 → 팀 승격 조건

3명 이상 디렉터에서 동일 패턴 발견 + result_score 평균 >= 0.7 일 때 팀 표준 후보로 제안됩니다.
장르 리드 디렉터가 검토 후 설계 표준에 반영 여부를 결정합니다.

## 기획 자료 분류 체계 (Base Design DB)

### 분류 목적

코드 Workflow에서 Base Code DB가 코드 재사용과 AI 참조에 활용되듯이,
Base Design DB는 AI가 기획 결과물을 생성할 때 직접 참조하는 데이터베이스입니다.
단순 체크리스트가 아니라, AI가 검색하여 활용하기 위한 구조입니다.

### 저장 구조: 2레이어

```
레이어 1: 프로젝트 단위 통째 저장 (맥락 보존)
  → "이 게임의 전체 기획이 어떤 구조였고 결과가 어땠는지" 참조
  → 프로젝트 간 비교 분석, 전체 구조 참조 시 사용

레이어 2: 세부 요소 단위 태깅 저장 (검색/비교용)
  → "데미지 공식만", "가챠 확률만", "성장 곡선만" 뽑아서 교차 비교
  → AI가 특정 기획 요소를 생성할 때 유사 사례를 빠르게 참조
  → 디렉터가 "쌓인 DB에서 데미지 공식 예시들 보여줘" 할 때 활용
```

레이어 1에서 레이어 2로의 추출은 AI가 자동 수행하고, 사람이 검수합니다.

### 분류 축

```
기획 도메인 > Genre > 시스템 단위
+ 다중 태그 (밸런스 영역, 데이터 유형, 성과 등급, 출처, 버전)
```

#### 기획 도메인 (기획자의 실제 업무 분업에 대응)

| 도메인 | 설명 | 포함 범위 예시 |
|--------|------|---------------|
| InGame | 인게임 시스템 기획 | 전투, 조작, 스킬, AI, 물리, 스테이지 구조, 전투 밸런스, 월드맵/영토(SLG), 보드/매칭(퍼즐) |
| OutGame | 아웃게임 시스템 기획 | 인벤토리, 장비, 가챠, 강화, 합성, 도감, 우편함 |
| Balance | 밸런스 기획 | 성장 곡선, 재화 밸런스, 난이도 곡선, 확률 설계 |
| Content | 콘텐츠 기획 | 스토리, 퀘스트, 이벤트, 던전, PvP, 시즌, 레벨디자인 |
| BM | 수익 모델 기획 | 과금 구조, 패키지, 시즌패스, 광고, 구독, VIP |
| LiveOps | 라이브 운영 기획 | 업데이트 주기, 이벤트 캘린더, 공지, 보상 정책. 코드 매핑: Config + Service + Scheduler Role |
| UX | UX/UI 흐름 기획 | 화면 전환, 튜토리얼, HUD, 팝업 정책, 유저 동선 |
| Social | 소셜 기획 | 길드, 친구, 채팅, 랭킹, 협동, 거래소 |
| Meta | 메타 시스템 기획 | 계정 성장, 도전과제, 컬렉션, 프레스티지 |

장르별 특수 시스템(SLG 월드맵 영토, 퍼즐 보드 시스템 등)도 위 도메인에 귀속시키고,
장르+시스템 태그로 세분화합니다. 기획자가 만드는 거의 모든 시스템은 위 도메인 안에 들어갑니다.

#### Genre

코드 Workflow와 동일:
```
Generic(공통) / RPG / Idle / SLG / Simulation / Tycoon / Merge / Puzzle / Casual
```

> **Playable은 장르가 아닌 플랫폼입니다.** CLAUDE.md의 `platform: playable` 분기 구조를 유지하며, 플레이어블 광고의 장르는 Casual, Puzzle 등 게임 메카닉에 따라 결정됩니다.

Generic 장르를 두어 장르 무관한 기획 패턴은 공통으로 관리하고,
검색 시 "해당 장르 → Generic 순서"로 참조하여 데이터 부족 문제를 방지합니다.

#### 복합 장르 처리 규칙

하나의 게임이 여러 장르 특성을 가질 수 있습니다:

```yaml
방식: 주 장르 1개 + sub_genre 배열
  genre: "Idle"
  sub_genre: ["RPG"]

DB 검색: 주 장르(genre)로 검색, sub_genre는 참고 태그
  → Idle DB에서 1차 검색
  → RPG DB에서 보조 참조 (매칭 가중치 낮음)
  → Generic DB에서 보완

기획서에서의 표기:
  game_overview.genre: "Idle"        # 주 장르 (1개)
  game_overview.sub_genre: ["RPG"]   # 부 장르 (복수 가능)
```

#### 다중 태그

하나의 기획 자료가 여러 관점에서 검색될 수 있으므로 다중 태그를 적용합니다.

```yaml
예시: "애쉬엔베일 데미지 공식"
  project: ash_n_veil
  domain: InGame
  genre: Idle RPG
  system: 전투 > 데미지 판정
  balance_area: 전투 밸런스          # 밸런스 관점 태그
  data_type: 공식/수치               # 데이터 유형 태그
  performance_grade: A               # 성과 등급 (사후 라벨)
  source: internal_original          # 출처 유형
  version: 1.0.0                     # 데이터 버전
```

#### source (출처 유형) 태그 — 6종

| source 값 | 설명 |
|-----------|------|
| internal_original | 자사 기존 프로젝트에서 입력한 완성된 자료 (실증 데이터) |
| internal_produced | Workflow 과정을 통해 생산되어 디렉터 검증을 거친 자료 |
| internal_live | 라이브 운영 중 수정/업데이트된 자료 (KPI 데이터 포함) |
| observed | AI Tester 관찰로 추출한 외부 게임 분석 자료 |
| community | 위키, 가이드 등 공개 커뮤니티 데이터 |
| generated | AI가 추론하여 생성한 보완 데이터 (역추론, 빈 부분 채우기 등) |

internal_original과 internal_produced를 구분하는 이유:
- internal_original은 "실제로 이렇게 했고 결과는 이랬다"는 실증 데이터
- internal_produced는 "AI가 만들고 디렉터가 검증한" 데이터
- 신뢰도의 성격이 다르므로, AI가 검색할 때 어떤 근거의 데이터인지 구분할 수 있어야 합니다

### DB 검색 우선순위

코드 Workflow의 DB 검색 우선순위와 동일한 구조:

```
Expert Design DB: 검증 완료 + 성과 확인된 기획 자료
Base Design DB: 정규화되었으나 성과 미확인 자료

순위    검색 대상                       조건
---------------------------------------------------
1       Expert DB (해당 장르)           genre 일치 AND score >= 0.6
2       Expert DB (Generic)             genre = Generic AND score >= 0.6
3       Genre Base DB                   genre 일치
4       Generic Base DB                 genre = Generic
5       AI 자체 생성                    참조 자료 없음 → 디렉터 상세 디렉션 필수
```

### 검수

코드 Workflow 1단계 검수와 동일한 구조:

```
중복 확인: 같은 내용이 다른 태그로 중복 저장되어 있는지
누락 확인: 기획서에 있는 시스템인데 DB에 추가 안 된 것이 있는지
태깅 정확성: 도메인, 장르, 시스템, 태그가 정확한 값을 가지고 있는지
맥락 보존: 세부 요소로 분리했을 때 원래 기획서의 맥락이 훼손되지 않았는지
```

## [1단계] Base Design DB 가공

### 의도

자사 기존 기획 자료, 외부 레퍼런스 게임 분석 자료, AI Tester 관찰 자료를
AI가 참조 가능한 정규화된 DB로 가공하여 축적합니다.

### 입력 자료의 현실적 상태와 처리

과거 프로젝트 자료는 형태가 제각각입니다. 이를 처리하는 방법:

| 입력 상태 | 처리 방법 |
|----------|----------|
| 완전한 기획서 + 밸런스 시트 존재 | AI가 분류 체계에 맞춰 자동 가공 → 사람 검수 |
| 밸런스 시트만 있고 시스템 기획서 없음 | AI가 테이블에서 시스템 규칙 역추론하여 기획서 초안 생성 → 사람 검수 |
| 기획서는 있으나 비정형 (PPT 개조식 등) | AI가 정규화된 형식으로 재구성 → 사람 검수 |
| 기획서와 실제 구현이 다름 | AI Tester 자사 게임 테스트로 실제 수치 추출 → 기획서와 교차 검증 |
| 외부 레퍼런스 게임 (소스 없음) | AI Tester 순수 관찰 분석 → Design Intent 수준이 아닌 관찰 데이터 그대로 활용 |

### 과정

```
1. 입력 자료 진단
   : 자료 형태 자동 분류 (완전/불완전/비정형/외부)

2. 정규화
   : 분류 체계에 맞춰 기획 도메인 > Genre > 시스템 단위로 정리
   : 다중 태그 자동 부여

3. 빈 부분 역추적 채우기
   : 밸런스 테이블 → 시스템 규칙 역추론
   : 소스코드 → 기획 의도 역추론 (코드 Workflow 1단계와 연결)
   : AI Tester 관찰 → 플레이 기반 시스템 구조 추출

4. 레이어 2 추출
   : 통짜 기획서(레이어 1)에서 세부 요소(레이어 2)를 추출하여 태깅

5. 설계 의도 분석 + 품질 평가
   : AI(Design DB Builder)가 파싱된 각 기획 요소에 대해 분석
   : design_intent  — 이 설계가 왜 이렇게 되었는지 (의도 추론)
   : context        — 맥락, 제약조건, 전제 조건
   : strengths      — 이 설계의 강점
   : concerns       — 잠재적 약점이나 리스크
   : db_recommendation — DB 투입 판단 (store / store_with_caveat / skip / needs_context)
   : reasoning      — 판단 근거

6. 디렉터 큐레이션
   : AI 분석 확인 — design_intent가 실제 의도와 맞는지, db_recommendation에 동의하는지
   : 맥락 보충   — 성과 데이터, 실패 원인, 임시 설계 여부 등 사람만 아는 정보 추가
   : 최종 판단   — 승인(DB 저장) / 수정 후 승인 / 거부(미저장, 사유 기록)
   : 기존 검수   — 중복, 누락, 태깅 정확성, 맥락 보존 확인

7. DB 저장
   : 큐레이션 승인분만 저장
   : 디렉터 보충 맥락이 있으면 해당 데이터에 반영
```

#### Step 5 상세: 설계 의도 분석 + 품질 평가

AI(Design DB Builder)가 파싱된 각 기획 요소에 대해 자동 분석을 수행합니다.
단순히 "파일 → DB 변환"이 아니라, "이 기획이 왜 이렇게 설계됐는지, DB에 넣을 만한 수준인지"를 판단합니다.

```yaml
design_analysis:
  design_intent: "방어력 데미지 감소 비중을 낮춰 공격적 플레이 유도"
  context: "감산식 데미지, 방어율 상한 75%로 탱커 메타 방지"
  strengths: ["공식 간결", "밸런스 조정 포인트 명확"]
  concerns: ["방어 투자 가치 낮을 수 있음", "PvP에서 원샷 위험"]
  db_recommendation: "store"
  reasoning: "검증된 프로젝트의 실증 데이터, 구조 참조 가치 있음"
```

db_recommendation 4종:

| 값 | 의미 | 디렉터 액션 |
|---|------|-----------|
| store | DB 투입 가치 있음 | 확인 후 저장 |
| store_with_caveat | 조건부 가치 (약점 있음) | caveat 확인 후 판단 |
| skip | DB 투입 부적합 | override 가능 |
| needs_context | 맥락 부족, 판단 불가 | 맥락 보충 필요 |

#### Step 6 상세: 디렉터 큐레이션

기존 "검수"(중복/누락/태깅/맥락)를 확장하여, 디렉터가 AI 분석 결과를 확인하고 최종 판단합니다.

```
디렉터 큐레이션 범위:
  1. AI 분석 확인
     - design_intent가 실제 의도와 맞는지
     - db_recommendation에 동의하는지

  2. 맥락 보충
     - "이건 리텐션 목적이었고 D7 25%였어" → performance context
     - "이건 실패한 설계야" → 원인 태그 + 낮은 점수
     - "이건 임시로 넣은 거야" → skip

  3. 최종 판단
     - 승인 → DB 저장
     - 수정 후 승인 → AI 수정 → 재확인
     - 거부 → 미저장 (사유 기록)

  4. 기존 검수 항목 유지
     - 중복, 누락, 태깅 정확성, 맥락 보존
```

### 자료 출처별 처리

#### 자사 기존 프로젝트
- 기획서, 밸런스 시트, 데이터 테이블을 직접 입력
- 소스코드가 있으면 코드 Workflow 1단계(Base Code DB)와 교차 참조하여 정확도 향상
- AI Tester 자사 게임 테스트를 병행하여 기획서와 실제 구현의 차이를 검출

#### 외부 레퍼런스 게임
- AI Tester 시스템으로 순수 관찰 분석 (AI_tester.md 참조)
- 소스코드 접근 없이 10명 AI 전문 관찰(AI Tester 표준 프로세스)만 사용
- 패턴 카드(Level 1/Level 2) 방식은 사용하지 않음
  - 이유: 패턴 카드는 소스코드 역추출 경로에서 생성되었을 가능성이 있어 법적 리스크 존재
  - AI Tester 10명 관찰만으로 약 85~89.5% 정확도 달성, Workflow 참조용으로 충분
  - 나머지 10~15%는 자체 설계로 채우며, 이것이 오히려 차별화 요소가 됨
- 32개 파라미터 추정 결과를 Base Design DB로 가공 (source: observed)

#### 커뮤니티/공개 데이터
- 위키, 가이드, 공시 확률 등 공개 정보 수집
- AI Tester Phase 2.5(WIKI 교차검증)와 연계

## [2단계] 기획 생성

### 의도

디렉터의 상위 디렉션과 Base Design DB를 기반으로,
AI가 기획서, 밸런스 시트, 콘텐츠 데이터를 생성합니다.

### 기획 도메인별 생성 프로세스

기획은 실제 현업에서 순차적으로 진행됩니다:
```
시스템 기획 → 콘텐츠 기획 → 밸런스 기획 → BM 기획 → 통합 검증
```

#### 기획 생성 병렬화 규칙

2단계 기획 생성 시, 일부 도메인은 병렬 실행이 가능합니다:

```
순차 필수:
  2-1. 컨셉 정의 (모든 후속 작업의 기반)
    ↓
  2-2. 시스템 기획 (콘텐츠/밸런스/BM의 기반)

병렬 가능:
  2-2 완료 후 →  2-3. 밸런스 기획  ─┐
                  2-4. 콘텐츠 기획  ─┤ 동시 진행 가능
                                    ↓
                  2-5. BM 기획     ← 밸런스 결과 참조 필요 (2-3 완료 후)
```

- 2-3(밸런스)와 2-4(콘텐츠)는 시스템 기획(2-2)만 완료되면 병렬 착수 가능
- 2-5(BM)는 경제 밸런스 수치에 의존하므로 2-3 완료 후 시작
- Agent Team 활용 시 Designer가 2-3과 2-4를 동시에 생성하여 속도 향상

각 단계에서 동일한 패턴이 반복됩니다:
```
디렉터 디렉션 → AI가 DB 검색 → AI가 결과물 생성 → 디렉터 검증
```

### 2-1. 컨셉 정의

디렉터 디렉션:
- 장르, 타겟 유저, 핵심 경험(디자인 필러), 레퍼런스 게임

AI 실행:
- 레퍼런스 게임 분석 보고서 생성 (AI Tester 관찰 결과 기반 또는 DB에서 검색)
- 디자인 필러와 핵심 루프의 일관성 체크
- 컨셉 문서 초안 작성

출력물:
```
컨셉 문서
├── 디자인 필러 (3~5개)
├── 핵심 루프 다이어그램
├── 타겟 유저 프로파일
├── 세션 구조 (1회 플레이 목표 시간, 일일 목표 시간)
├── 레퍼런스 분석 요약
└── 차별화 포인트
```

### 2-2. 시스템 기획

디렉터 디렉션:
- 각 시스템의 상위 규칙과 존재 이유
- 시스템 간 우선순위와 의존 관계
- "이 부분은 레퍼런스와 다르게 이렇게 가자" 수준의 방향

AI 실행:
1. 시스템 목록 자동 도출: 컨셉 + 장르 기반으로 필요 시스템 전체 제안
2. 기획서 구조 채우기: 디렉터의 상위 규칙을 기반으로 세부 명세 작성
   - DB에서 유사 시스템 검색하여 참조
   - 규칙 상세, 데이터 테이블 구조, 예외 처리, UI 플로우 포함
3. 시스템 간 일관성 체크
   - 시스템 A의 규칙과 시스템 B의 규칙 사이 충돌 검출
   - 순환 의존성 검출
4. UI 플로우 차트 자동 생성

출력물:
```
시스템 기획서 (시스템별)
├── 시스템 개요 (목적, 해결하는 문제)
├── 규칙 정의
├── 데이터 테이블 구조
├── UI 플로우
├── 예외/엣지 케이스
└── 연결 시스템 목록

시스템 연결 관계도 (전체)
일관성 검증 보고서
```

### 2-3. 밸런스 기획

디렉터 디렉션:
- 밸런스 의도 ("Lv.10까지 3일, 초반은 레퍼런스보다 쉽게, 중후반 난이도 상승")
- 핵심 수치 방향 ("감산식 데미지, 방어 비중 낮게")
- 과금 압박 수준 ("Day 7 첫 번째 과금 포인트")

AI 실행:
1. 수치 테이블 생성
   - DB에서 유사 장르/시스템의 밸런스 데이터 검색
   - 디렉션에 맞춰 전체 테이블 생성 (레벨별 스탯, 경험치, 비용 등)
2. 성장 곡선 설계
   - 공식 역산 및 시각화
3. 경제 시뮬레이션
   - 재화 Source/Sink 모델링
   - 무과금/소과금/고래 유저 세그먼트별 30일 시뮬레이션
   - 인플레이션 발생 시점 예측
4. 전투 시뮬레이션
   - 캐릭터/무기 간 DPS 비교
   - 특정 조합이 지배적(OP)인지 검출
5. 확률 설계
   - 가챠 기대값 계산 (평균 뽑수, 비용 분포)
   - 강화 시뮬레이션 (평균 시도 횟수, 재화 소모량)

출력물:
```
밸런스 시트
├── 성장 테이블 (레벨별 스탯, 경험치, 비용)
├── 전투 공식 정의서
├── 재화 흐름도 (Source/Sink 다이어그램)
├── 확률 테이블 (가챠, 강화, 드랍)
└── 경제 시뮬레이션 보고서
    ├── 무과금 유저 30일 시뮬레이션
    ├── 소과금 유저 30일 시뮬레이션
    └── 고래 유저 30일 시뮬레이션
```

### 2-4. 콘텐츠 기획

디렉터 디렉션:

- 유저 감정 곡선 ("5챕터에서 반전, 10챕터에서 클라이맥스")
- 이벤트 기획 방향

AI 실행:
1. 데이터 스키마 로드
   - 0단계 설계 표준에서 해당 콘텐츠의 데이터 스키마 참조
   - 스키마 미정의 시 디렉터+프로그래머 싱크 요청 (0단계로 되돌림)
2. 콘텐츠 양산
   - 디렉터가 "타입 A 몬스터" 규칙 정의 → AI가 100종 스탯/스킬/드랍 테이블 생성
   - Phase 2-3 밸런스 시트 참조하여 수치 자동 맞춤
   - 데이터 스키마에 맞춰 실제 JSON/YAML 데이터 생성 (바로 사용 가능한 형태)
3. 스테이지 구성 초안
   - 난이도 곡선에 맞춰 스테이지별 몬스터 배치, 보상 배분
4. 퀘스트 체인 일관성 체크
   - 선행 조건 오류, 보상 중복, 미해금 아이템 참조 등 논리 오류 검출
5. 이벤트 영향도 분석
   - "이 이벤트를 열면 재화 밸런스에 어떤 영향을 주는가" 시뮬레이션
6. 데이터 테이블 생성
   - 기획 의도를 서버/클라이언트에 들어갈 데이터 테이블 형식으로 변환
   - 0단계 데이터 스키마 + enum 정의에 맞춰 유효성 자동 검증

## [3단계] 통합 검증

### 의도

2단계에서 생성된 기획 결과물 전체가 하나의 게임으로서 일관되는지 검증합니다.

### 과정

#### 3-1. 교차 일관성 체크

```
시스템 ↔ 밸런스: 시스템 규칙과 밸런스 수치가 일치하는가
콘텐츠 ↔ 시스템: 콘텐츠에서 참조하는 아이템/스킬이 시스템에 정의되어 있는가
BM ↔ 밸런스:     BM에서 파는 아이템이 밸런스를 파괴하지 않는가
UX ↔ 시스템:     UI 플로우에서 누락된 화면이나 접근 불가 기능이 없는가
```

#### 3-2. 유저 여정 시뮬레이션

Day 1~30 유저 경험을 시뮬레이션합니다:
```
Day 1: 튜토리얼 → 1-1~1-5 클리어 → Lv.5 → 첫 가챠 → ...
Day 3: Lv.10 → 장비 강화 해금 → 첫 보스전 → ...
Day 7: 첫 번째 과금 포인트 도달 → ...
```
각 시점에서 유저가 접하는 시스템, 콘텐츠, 재화 상태를 추적하여 기획 의도와 비교합니다.

#### 3-3. 누락 검출

```
"전투에서 '치명타 저항' 스탯을 쓰는데, 이걸 올릴 수 있는 수단이 기획에 없습니다"
"튜토리얼에서 장비 강화를 가르치는데, 강화 재료를 얻는 콘텐츠가 Lv.15 이후에만 열립니다"
"상점에 '전설 강화석'이 있는데, 시스템 기획에 이 아이템의 효과가 정의되지 않았습니다"
```

#### 3-4. AI 자가 검증 결과 주석 추가

코드 Workflow 5단계와 동일한 원리로, AI가 검증 과정에서 발견한 사항을 주석으로 추가합니다.
주석이 많은 기획서는 피드백을 많이 받은 것이므로 신뢰도 지표로 활용됩니다.

## [4단계] 디렉터 검수 및 피드백

### 의도

디렉터(사람)가 AI가 생성한 기획 결과물을 검수하고, 피드백을 제공합니다.
Expert Design DB에 품질 높은 기획 자료를 축적하기 위한 과정입니다.

### 과정

```
1. 기획 리뷰
   디렉터가 결과물을 확인하며 피드백 사항 정리
   피드백이 없을 경우 → [6단계] 진행
   피드백이 있을 경우 → 피드백 작성 진행

2. 피드백 작성
   피드백 형식에 맞춰 정리한 사항 작성

3. 기획 재생성
   피드백 자료를 기반으로 해당 부분만 재생성
   (피드백 DB가 충분히 축적된 후에는 자연어 피드백 → AI가 정해진 포맷으로 자동 변환)
```

### 피드백 형식

코드 Workflow의 피드백 형식에 대응:

| 항목 | 설명 | 예시 |
|------|------|------|
| targetId | 피드백 대상 | Balance.Growth.LevelTable |
| domain | 기획 도메인 | Balance |
| genre | 장르 | RPG |
| category | 피드백 유형 | BALANCE.CURVE_TOO_STEEP |
| section | 대상 섹션 (선택) | Lv.20~30 구간 |
| description | 피드백 설명 | 20~30 구간 경험치 증가가 너무 급격함. 유저 이탈 우려 |
| direction | 수정 방향 | 해당 구간 곡선을 완만하게 조정, 레퍼런스 A 게임 수준으로 |
| designPillarCheck | 디자인 필러 위반 여부 | false |

### 피드백 카테고리

```
SYSTEM.RULE_CONFLICT       시스템 규칙 간 충돌
SYSTEM.MISSING_FEATURE     누락된 기능
SYSTEM.OVER_COMPLEXITY     불필요하게 복잡한 구조
BALANCE.CURVE_TOO_STEEP    성장/난이도 곡선 과도
BALANCE.CURVE_TOO_FLAT     성장/난이도 곡선 부족
BALANCE.ECONOMY_IMBALANCE  재화 생산/소비 불균형
BALANCE.FORMULA_ERROR      수식 오류
CONTENT.PACING_ISSUE       콘텐츠 배분 문제
CONTENT.LOGIC_ERROR        퀘스트/스테이지 논리 오류
BM.PAY_WALL_TOO_HARD       과금 벽 과도
BM.VALUE_MISMATCH          가성비 불일치
UX.FLOW_BROKEN             유저 동선 단절
UX.TUTORIAL_GAP            튜토리얼 누락 구간
DIRECTION.OFF_TARGET       디자인 필러/컨셉과 불일치
```

## [5단계] 재생성 기획 평가

### 의도

피드백을 통해 재생성한 기획의 품질을 평가하여,
파이프라인 자체의 문제를 발견하고 Expert Design DB에 축적할 기획을 선별합니다.

### 과정

```
1. 재생성 결과 검증
   AI가 피드백 사항이 정확히 반영되었는지 확인 후 주석 추가
   - 반영 완료 → [6단계] 진행
   - 미반영 또는 새로운 이슈 → 이슈 목록 정리 후 재생성

2. 피드백 히스토리 분석
   재생성 과정에서 반복적으로 발생하는 피드백 패턴 정리
   → 지속적으로 발생하는 이슈나 잠재적 이슈에 대한 예방 규칙 도출

3. 이전 버전과의 차이점 기록
   기획 변경 히스토리를 남겨 추후 동일한 이슈 발생 시 참조
```

## [6단계] 평가 및 DB 축적

### 의도

검증 완료된 기획 자료를 Expert Design DB에 축적하여,
AI가 참조할 품질 높은 DB를 구축합니다.

### DB 물리 구조

```
E:\AI\db\design\
├── base\
│   ├── {genre}\                  # 장르별 디렉토리 (generic, rpg, idle, ...)
│   │   ├── _projects\            # ← Layer 1: 프로젝트 통짜 저장
│   │   │   └── {project}.json    #    프로젝트 전체 기획 구조 1파일
│   │   ├── {domain}\             # ← Layer 2: 세부 요소 단위
│   │   │   ├── index.json        #    검색용 인덱스 (경량)
│   │   │   └── files\
│   │   │       └── {designId}.json  # 상세 데이터
│   │   └── ...
│   └── ...
├── expert\                       # 검증 완료 (score >= 0.6)
│   ├── index.json
│   └── files\
│       └── {designId}.json
└── rules\                        # 축적된 피드백 규칙
    ├── generic_design_rules.json  # 장르 무관
    ├── genre_design_rules.json    # 장르별
    └── domain_design_rules.json   # 도메인별
```

- Layer 1(`_projects/`): 프로젝트 전체 기획 구조를 1파일에 보존. 프로젝트 간 비교, 전체 구조 참조 시 사용
- Layer 2(`{domain}/files/`): 세부 요소별 태깅. AI가 특정 기획 요소를 빠르게 검색/참조할 때 사용

### 기획 신뢰도 점수 체계

코드와 기획의 근본적 차이를 반영한 점수 체계입니다:
- 코드: 에러 없으면 100% 품질, 재활용 가능
- 기획: 에러 여부가 아니라 재밌냐, 성과 났냐, 돈 벌었냐가 중요

#### 점수 체계 통합 정의

기획의 점수 체계는 **자동 점수 3종**(AI가 산출)과 **신뢰도 점수**(이벤트 기반 변동) 2단계로 구성됩니다.

**자동 점수 3종 (AI 판단, 각 0~1.0)**

| 항목 | 설명 | 산출 기준 |
|------|------|-----------|
| 논리 완결성 | 시스템 간 모순, 누락, 참조 오류 없을수록 높음 | Cross-Consistency + Gap Detection 결과 |
| 밸런스 안정성 | 경제/전투 시뮬레이션에서 이상치 없을수록 높음 | balance-simulator.js 결과 |
| 구현 복잡도 | 기존 Base Code DB와 매칭률 (구현 가능성) | DB 검색 매칭 비율 |

**자동 점수 → 신뢰도 초기값 결정**

```
자동 점수 3종 평균값 계산:
  → 평균 >= 0.5 → 신뢰도 초기값 0.4 부여
  → 평균 <  0.5 → 신뢰도 초기값 0.3 (검증 미달)
  → Expert DB 승격은 신뢰도 점수 >= 0.6 기준 (자동 점수는 초기값만 결정)
```

#### 신뢰도 점수 변동

| 이벤트 | 점수 변동 |
|--------|----------|
| 초기 저장 (자동 점수 평균 >= 0.5) | 0.4 |
| 초기 저장 (자동 점수 평균 < 0.5) | 0.3 |
| 디렉터 검증 통과 (피드백 없이 승인) | +0.2 |
| 피드백 반영 완료 후 승인 | +0.1 |
| 다른 프로젝트에서 구조 참조 성공 | +0.1 |
| 참조 부적합 판정 | -0.1 |
| 다른 장르에서 참조 성공 | +0.1 (Generic 승격 검토) |
| Expert Design DB 승격 임계값 | >= 0.6 |

> 기획 재사용 실패 감점(-0.1)은 코드(-0.15)보다 완화됨. 근거: 기획 재사용 실패는 코드보다 원인이 복잡하므로(장르 특성, 타겟 유저, 시장 상황 등) 감점폭을 완화.

> 감점은 자동 적용이 아닌, 디렉터가 명시적으로 "참조 부적합"을 마킹한 경우에만 적용합니다.

#### 사후 라벨링 (출시 후, 사람이 기록)

| 항목 | 설명 |
|------|------|
| 시장 성과 등급 | S / A / B / C / F (매출, 리텐션, DAU 종합) |
| 원인 분석 태그 | 아는 만큼만 기록 (선택) |

사후 라벨은 신뢰도 점수에 직접 반영하지 않고 별도 필드로 관리합니다.
이유: 시장 성과는 기획 외 요인(마케팅, 출시 타이밍 등)에 영향받으므로,
기획 자체의 품질 점수와 혼합하면 DB 오염이 발생합니다.

#### performance_grade 필드 스키마 (사후 라벨 저장 형식)

```yaml
performance_grade:
  market_grade: "B"                      # S/A/B/C/F
  metrics:
    d1_retention: 0.45
    d7_retention: 0.22
    d30_retention: 0.08
    arpu_30d: 2.5
    dau_peak: 15000
  cause_tags: ["weak_endgame", "no_social"]  # 원인 분석 (선택)
  labeled_by: "director"                 # 라벨 작성자
  labeled_at: "2026-03-15"               # 라벨 작성일
```

#### 사후 라벨 기반 검색 쿼리

DB에 사후 라벨이 축적되면, 성과 기반 검색이 가능해집니다:

```
검색 예시:
  "Idle 장르에서 D7 리텐션 30% 이상인 프로젝트의 경제 밸런스"
  → genre=Idle, performance_grade.metrics.d7_retention >= 0.3, domain=Balance

  "가챠 천장이 있는 RPG 중 ARPU 상위 프로젝트"
  → genre=RPG, tags 포함 "gacha_pity", 정렬: performance_grade.metrics.arpu_30d DESC

  "성과 A등급 이상 프로젝트의 튜토리얼 설계"
  → performance_grade.market_grade IN ["S", "A"], domain=UX, tags 포함 "tutorial"
```

### DB 분기

```
신뢰도 점수 >= 0.6 → Expert Design DB에 저장
신뢰도 점수 <  0.6 → 저장하지 않음 (Base DB에만 유지)
```

### 실패 프로젝트 데이터 처리

성과가 덜한 프로젝트의 데이터는 목적에 따라 구분하여 처리합니다:

| 활용 목적 | 처리 방법 | 조건 |
|----------|----------|------|
| 구조/포맷 참조 | 항상 활용 | 성패 무관. 테이블 형식, 시스템 연결 구조, 공식 형태 등 |
| 수치 참조 | 조건부 활용 | 사후 라벨 있는 경우만. "이 수치에서 리텐션 B등급" 수준의 맥락 제공 |
| 반면교사 | 원인 명확할 때만 | "가챠 천장 없어서 유저 이탈" 등 원인 태그가 있는 경우만 |

"왜 안 됐는지 원인을 모르는 경우"에는 반면교사로 사용하지 않습니다.
원인 불명 상태에서 "이렇게 하지 마"를 학습시키면 AI가 잘못된 규칙을 학습할 수 있습니다.

### Rules 추출

코드 Workflow 6단계와 동일 구조:

| 규칙 유형 | 설명 | 예시 |
|----------|------|------|
| Generic 규칙 | 장르 무관 공통 규칙 | "재화 Source 없이 Sink만 있는 구간 3일 이상 금지" |
| 장르별 규칙 | 특정 장르에만 적용 | "Idle RPG 방치보상 최대 축적 8~24시간 권장" |
| 도메인별 규칙 | 특정 기획 도메인에 적용 | "가챠 천장 미설정 시 유저 이탈 위험 경고" |

## [7단계] 플레이 검증

### 의도

문서 레벨의 검증(3단계)을 넘어, 실제 플레이 기반으로 기획 품질을 실증합니다.
코드 Workflow의 결과물이 구현된 이후에 진행됩니다.

### 코드 Workflow와의 연결

```
기획 Workflow 3단계(통합 검증) 통과
    ↓ 기획서를 코드 Workflow 2단계(시스템 명세서)로 자동 변환
코드 Workflow 2~6단계 진행
    ↓ 플레이 가능한 빌드 생성
기획 Workflow 7단계(플레이 검증) 진행
```

### 7-1. 자사 게임 가속 테스트

AI Tester 인프라(BlueStacks + ADB)를 그대로 활용합니다.

```
방법:
  서버 통신 끊고 10~20배속 가속 환경에서 AI Tester가 플레이
  30일치 플레이를 반나절에 완료

검증 항목:
  밸런스 시트 예측값 vs 실제 플레이 수치 비교
  기획 의도 vs 실제 유저 경험 괴리 확인
  병목 구간 발견 (진행 막히는 지점, 재화 부족 구간)

결과:
  차이가 큰 부분 → 기획 Workflow 2단계로 되돌아가 밸런스 수정
  차이가 작은 부분 → 기획 확정, Expert Design DB에 실증 데이터 추가
```

### 7-2. 외부/방치형 게임 장기 테스트

방치형 게임처럼 하루이틀 플레이로는 전체를 파악할 수 없는 경우:

```
1일차: AI가 관찰 가능한 데이터로 30일치 예측 모델 생성
       → 이 예측으로 기획 2단계 밸런스 설계에 즉시 반영 (착수 가능)

1~30일차: AI Tester 병렬로 계속 실제 플레이
          → 매일 예측 vs 실제 차이 누적

30일차: 예측치와 달라진 부분만 사후 보정
        → 보정된 데이터를 Base Design DB에 저장
```

가속화 방법 (법적 문제 없는 범위):
- 자사 게임: 서버 가속, 시간 스킵 API 사용
- 외부 게임: 실시간 플레이만 가능 → 예측 선행 + 사후 보정 방식으로 대응

### 7-3. AI 가상 유저 대규모 테스트

AI Tester 관찰 정확도가 90%+ 달성되면, AI 가상 유저 에이전트를 생성하여
천~만 명 규모 동시 시뮬레이션이 가능해집니다.

```
활용 예시:

퍼즐 게임:
  1000명 가상 유저 × 100 스테이지
  → 스테이지별 클리어율, 실패 지점, 평균 시도 횟수 분포
  → 레벨 디자인 피드백 자동 생성

SLG:
  1000명 가상 유저 × 서버 2주 시뮬레이션
  → 자원 병목, 전력 격차, 이탈 시점 발견
  → 밸런스 수정 포인트 자동 도출

RPG:
  무과금 70% / 소과금 25% / 고래 5% 비율로 시뮬레이션
  → 과금 벽 체감 지점 확인
  → BM 기획 피드백

결과 활용:
  발견된 병목 패턴을 DB에 기록
  → 다음 프로젝트의 기획 2단계에서 "이런 구간에서 병목이 자주 발생" 참조
  → 장르 무관 공통 패턴은 Generic 규칙으로 Rules에 추가
```

#### 7-3 스케일업 로드맵

AI 가상 유저 대규모 테스트의 단계적 확장 계획:

```
Phase A (MVP): 10명 가상 유저
  - 단일 장르, 핵심 루프만 검증
  - AI Tester 기본 행동 모델 사용
  - 목표: 시스템 동작 검증, 밸런스 이상치 탐지

Phase B (확장): 100명 가상 유저
  - 다양한 플레이 스타일 페르소나 적용
  - 과금/무과금 비율 설정 가능
  - 목표: 유저 세그먼트별 경험 차이 분석, 병목 구간 통계

Phase C (대규모): 1000명+ 가상 유저
  - 서버 부하 시뮬레이션 포함
  - 실시간 경제 시뮬레이션 (유저 간 거래, 경매)
  - 목표: 라이브 운영 사전 검증, 경제 안정성 실증
```

### Workflow 간 역방향 피드백 경로

7단계 플레이 검증에서 이슈 발생 시, 수정 범위에 따라 되돌아가는 경로가 다릅니다:

```
이슈 유형별 피드백 경로:

밸런스 이슈 (수치만 문제):

  → 예: "D15 이후 골드 소진률 120% 초과" → 경제 밸런스만 조정

시스템 이슈 (설계 문제):
  → 기획 2-2단계(시스템 기획) 수정
  → 코드: 해당 Phase부터 재시작
  → 예: "스킬 조합 시 무한루프 발생" → 스킬 시스템 재설계

양쪽 이슈 (기획+코드 모두):
  → 기획 2단계 수정
  → 코드: 전체 재생성
  → 예: "핵심 루프 자체가 재미없음" → 컨셉부터 재검토
```

### AI Tester 출력 → Design DB 변환 규칙

AI Tester의 관찰 결과를 Design DB에 자동 매핑하는 규칙:

| AI Tester 출력 | Design DB 필드 | Domain |

| 화면 구성 관찰 | UX flow 데이터 | UX |

| 재화 흐름 관찰 | economy source/sink | Balance |
| 가챠 확률 관찰 | gacha probability table | OutGame |
| 스테이지 구성 | stage content_data | Content |

| 상점 가격 관찰 | IAP/package config | BM |
| 소셜 기능 관찰 | social system rule | Social |

| AI 행동 패턴 | enemy AI rule | InGame |

```
변환 프로세스:
  AI Tester 관찰 데이터 (JSON)
    → design-parser.js --source-type observed
    → Base Design DB에 저장 (source: observed, reference_only: true)
    → 디렉터 검수 후 reference_only 해제 가능
```

## [8단계] 라이브 운영 DB 동기화

### 의도

런칭 후 라이브 운영 과정에서 기획/밸런스 수정이 계속 발생합니다.
특히 밸런스 테이블은 유저 반응, 매출 데이터, 메타 변화에 따라 빈번하게 변경됩니다.
이 변경사항이 DB에 반영되지 않으면, DB의 데이터가 현실과 괴리되어 다음 프로젝트에서 잘못된 참조가 발생합니다.

### 버전 관리 구조

```yaml
예시: 애쉬엔베일 데미지 공식 버전 히스토리

ash_n_veil/InGame/전투/데미지_공식:
  versions:
    - version: 1.0.0
      phase: pre_launch
      source: internal_produced
      data: "ATK × 배율 - DEF × 0.5, 최소 데미지 = ATK × 5%"
      note: "Workflow 2단계에서 생성, 디렉터 확정"

    - version: 1.1.0
      phase: launch
      source: internal_live
      data: "ATK × 배율 - DEF × 0.4, 최소 데미지 = ATK × 8%"
      note: "CBT 피드백 반영, 방어 비중 하향 조정"

    - version: 1.2.0
      phase: live_ops
      source: internal_live
      data: "ATK × 배율 - DEF × 0.35, 최소 데미지 = ATK × 10%"
      note: "시즌2 밸런스 패치, 무과금 유저 진행 속도 개선"
      live_context:
        trigger: "Day 14 리텐션 40% → 35% 하락"
        kpi_before: { d14_retention: 0.35 }
        kpi_after: { d14_retention: 0.42 }
```

### 과정

```
1. 라이브 패치 발생
   밸런스 수정, 시스템 변경, 신규 콘텐츠 추가 등

2. 변경사항 DB 반영
   기존 데이터를 덮어쓰지 않고, 새 버전으로 추가
   변경 이유(trigger)와 변경 전후 KPI를 함께 기록
   source: internal_live 태그 부여

3. 현행 버전 표시
   DB 검색 시 기본값은 최신 버전(현재 라이브 상태)
   과거 버전은 "이 수치에서 이런 결과가 나왔다"는 히스토리로 활용

4. 시즌/대규모 패치 단위 스냅샷
   대규모 밸런스 패치 시점에 전체 밸런스 시트의 스냅샷을 저장
   "시즌 1 전체 밸런스 상태"를 하나의 레이어 1 데이터로 보존
```

### 라이브 데이터 수집 자동화 수준

| 수준 | 설명 | 트리거 | 사람 개입 |
|------|------|--------|----------|
| L1 수동 | 사람이 변경사항을 직접 DB에 입력 | 수동 실행 | 전체 |
| L2 반자동 | 데이터 테이블 변경 감지 → AI가 DB 업데이트 초안 생성 → 사람 승인 | 파일 변경 감지 | 승인만 |
| L3 자동 | CI/CD 파이프라인에 연동, 배포 시 자동 DB 반영 + KPI 자동 수집 | 배포 이벤트 | 이상 시만 |

현재 MVP는 L1(수동) 수준이며, L2 → L3 순서로 자동화를 확장합니다.

### 버전 관리가 주는 가치

```
다음 프로젝트에서 참조할 때:

단순 참조: "이 게임의 데미지 공식" → 최신 버전 (v1.2.0)
히스토리 참조: "이 게임이 데미지 공식을 어떻게 바꿔왔는지" → 전체 버전 히스토리
인과 참조: "리텐션이 떨어졌을 때 어떤 밸런스 수정을 했는지" → trigger + KPI 데이터
```

이 데이터가 축적되면 AI가 "비슷한 상황에서 다른 프로젝트는 어떻게 대응했는지"를
구체적 KPI 변화와 함께 제시할 수 있게 됩니다.

### 라이브 운영 데이터의 DB 분류

| 변경 유형 | 처리 |
|----------|------|
| 밸런스 수치 수정 (공식, 테이블) | 해당 레이어 2 데이터에 새 버전 추가 |
| 신규 시스템 추가 | 새로운 레이어 2 데이터 생성 (source: internal_live) |
| 시스템 규칙 변경 | 해당 시스템 기획서에 새 버전 추가 |
| 콘텐츠 추가 (이벤트, 시즌) | 새로운 레이어 2 데이터 + 레이어 1 스냅샷 업데이트 |
| BM 변경 (패키지, 가격) | 해당 BM 데이터에 새 버전 추가 + 매출 KPI 기록 |

### 라이브 데이터의 신뢰도

라이브 운영에서 검증된 데이터는 가장 높은 신뢰도를 가집니다:
- 실제 유저가 플레이한 결과이므로 "체감 품질"이 실증됨
- KPI 데이터가 함께 기록되므로 "시장 성과"까지 확인 가능

다만, 라이브 데이터가 무조건 좋다는 뜻은 아닙니다:
- 급한 핫픽스로 밸런스를 깨뜨린 경우도 있음
- KPI 변화가 밸런스 수정 때문인지 다른 요인 때문인지 분리가 어려운 경우도 있음
- 따라서 라이브 데이터의 신뢰도는 기존 점수 체계와 별도로, KPI 데이터 유무와 인과관계 명확성으로 평가

## 전체 파이프라인 흐름도

```
[0단계] 장르별 설계 표준 정의 (장르 최초 1회 + 상시 보완)
    ├── 구성요소 맵 + 파라미터 정의 + DB 스키마
    ├── 디렉션 히스토리 자동 축적 시작
    └── 저장: db/design/standards/{genre}.yaml

[기존 기획 자료 / AI Tester 관찰 자료 / 레퍼런스 분석]
    ↓ 1단계 AI: 정규화 + 분류 + 태깅 + 설계 의도 분석 + 품질 평가 (0단계 스키마 참조)
    ↓ 디렉터 큐레이션: AI 분석 확인 + 맥락 보충 + 승인/거부
[Base Design DB]
    ├── 레이어 1: 프로젝트 단위 통짜
    └── 레이어 2: 세부 요소 단위 태깅

[디렉터 상위 디렉션]
    ↓ 2단계 AI: DB 검색 → 기획 생성
[기획 결과물]
    ├── 컨셉 문서
    ├── 시스템 기획서
    ├── 밸런스 시트 + 시뮬레이션
    ├── 콘텐츠 데이터
    └── BM 설계서

    ↓ 3단계 AI: 통합 검증
[검증 보고서]
    ├── 교차 일관성 체크 결과
    ├── 유저 여정 시뮬레이션
    └── 누락 검출 목록

    ↓ [4단계] 디렉터 검수
    ├── 피드백 없음 → [6단계]
    └── 피드백 있음 → 피드백 작성 → AI 재생성

    ↓ [5단계] 재생성 평가
    ├── 반영 완료 → [6단계]
    └── 이슈 잔존 → 재생성 반복

    ↓ [6단계] DB 축적
[Expert Design DB]
    ├── 신뢰도 >= 0.6 → Expert DB 저장
    ├── 신뢰도 <  0.6 → Base DB만 유지
    └── Rules 추출 → 반복 패턴을 조건부 규칙으로 저장

    ↓ 코드 Workflow 연결
[코드 Workflow 2단계] ← 기획서 → 시스템 명세서 자동 변환
    ↓ 코드 생성 → 빌드

    ↓ [7단계] 플레이 검증
[플레이 검증 결과]
    ├── 자사 가속 테스트 결과
    ├── 외부 장기 테스트 결과
    └── 가상 유저 대규모 시뮬레이션 결과
        ↓
    Expert Design DB에 실증 데이터 추가
    → 차이가 큰 부분은 2단계로 되돌아가 수정

    ↓ [8단계] 라이브 운영 DB 동기화
[라이브 운영]
    ├── 밸런스 패치 → DB에 새 버전 추가 + KPI 기록
    ├── 시스템 변경 → 기획서 새 버전 추가
    ├── 신규 콘텐츠 → 새 데이터 생성 + 스냅샷
    └── BM 변경 → BM 데이터 새 버전 + 매출 기록
        ↓
    축적된 라이브 데이터 → 다음 프로젝트의 1단계 입력으로 순환
```

## 기획 품질 게이트 (Quality Gates)

IdleMoney 프로젝트 실전 검증에서 발견된 6개 구조적 결함을 방지하기 위한 자동 검증 게이트입니다.
기획 3단계(통합 검증) 시 Design Validator가 아래 게이트를 자동 실행합니다.

### Gate 1: Cross-Layer Naming Validation (명칭 일관성)
```
검증 대상: L1(게임 기획서) / L2(시스템 명세서) / L3(AI_기획서 노드)에서
         동일 시스템을 지칭하는 명칭이 정확히 일치하는지 자동 검증

예시:
  L2 시스템명: "DataTableManager"
  L3 contract.requires: "DataTableManager"  ← 일치해야 함
  ✗ "DataManager" / "TableManager" 등 변형 사용 시 경고

발견 배경: IdleMoney에서 L2와 L3 간 시스템명 불일치 다수 발견
```

### Gate 2: L3 Completeness Gate (노드 완성도)
```
검증 대상: AI_기획서 노드(L3)의 존재 완성도

규칙:
  - Phase 0 + Phase 1 시스템의 L3 노드가 100% 존재해야 코드 생성 진행 가능
  - 전체 Domain 중 최소 80% L3 노드 존재해야 Validator 검증 요청 가능

발견 배경: IdleMoney에서 L3 노드 45% 커버리지로 코드 생성 시도 → 대량 누락
```

### Gate 3: Dependency Auto-Validation (의존성 자동 검증)
```
검증 대상: L3 노드의 dependencies.internal과 L2의 relations.uses 일치 여부

규칙:
  - L3에서 선언한 의존성이 L2에도 정의되어 있어야 함
  - L2에서 정의한 관계가 L3에도 반영되어야 함
  - 불일치 시 자동 목록 생성 → Designer에게 수정 요청

발견 배경: IdleMoney에서 L2↔L3 의존성 불일치 12건 발견
```

### Gate 4: logicFlow Quality Gate (로직 흐름 품질)
```
검증 대상: 각 L3 노드의 logicFlow

규칙:
  - 각 logicFlow에 최소 1개의 failNext/error 분기 존재 필수
  - 외부 의존(네트워크, 파일, DB) step에는 반드시 fallback step 정의
  - 무한 루프 가능성 있는 흐름 경고

발견 배경: 에러 처리 없는 logicFlow → 코드 생성 시 예외 처리 누락
```

### Gate 5: Copy-Paste Detection (복붙 탐지)
```
검증 대상: L3 노드 간 codeHints/avoidPatterns 중복

규칙:
  - codeHints/avoidPatterns가 3개 이상 노드에서 동일하면 경고
  - 시스템별 고유 패턴 최소 1개 필수
  - 복사-붙여넣기로 인한 시스템별 특성 누락 방지

발견 배경: IdleMoney L3에서 codeHints가 대부분 동일한 템플릿 → 코드 품질 저하
```

### Gate 6: Cross-Doc Consistency (교차 문서 일관성)
```
검증 대상: L1/docs/L2/L3 전체에서 동일 개념의 표기 일관성

규칙:
  - 등급명: L1 "Epic" = L2 "Epic" = L3 "Epic" (변형 "에픽", "EPIC" 경고)
  - 재화명: L1 "Diamond" = L2 "Diamond" = L3 "Diamond"
  - 시스템명: 전체 문서에서 동일 표기 유지
  - 수치: 동일 값이 여러 곳에 정의된 경우 불일치 경고

발견 배경: IdleMoney에서 재화명/등급명 불일치로 코드 생성 시 혼란
```

## 기획 도메인 → 코드 Domain 매핑

기획 결과물을 코드 Workflow로 넘길 때의 자동 변환 관계:

| 기획 도메인 | 코드 Domain |
|------------|-------------|
| InGame | Battle, Skill, Character, Stage |
| OutGame | Inventory, Shop, Item |
| Balance | 각 Domain의 Calculator/Processor Role로 매핑 |
| Content | Quest, Stage, Reward |
| BM | Shop, IAP |
| LiveOps | Config + Service + Scheduler Role |
| UX | UI, Audio |
| Social | Network, Guild |
| Meta | Achievement, Collection |

Balance 도메인은 코드에서 독립 Domain이 아니라 각 시스템의 Calculator/Processor Role로 매핑됩니다.
LiveOps 도메인은 코드에서 독립 Domain이 아니라 Config(설정값 관리) + Service(서버 통신) + Scheduler(주기 실행) Role로 매핑됩니다.
이는 기획과 코드의 구조적 차이를 반영한 것입니다.
