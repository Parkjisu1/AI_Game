## 1. Agent 팀 구조와의 대응

### 1-1. 기획 Workflow 3-AI vs 실제 Agent 6+1명

**기획 문서의 3-AI 분리** (기획 L18-28):
```
1단계 AI: Base Design DB 가공
2단계 AI: 기획서/밸런스/콘텐츠 생성
3단계 AI: 기획 검증 및 시뮬레이션
```

**실제 Agent 팀** (WORKFLOW.md L37-68, `.claude/agents/`):

| Agent | Model | 현재 역할 | 기획 Workflow 대응 |
|-------|-------|----------|-------------------|
| Lead | Opus | PM, 태스크 분배 | 기획 디렉터 지원? |
| Designer | Sonnet | 기획서 생성 (Layer 1/2/3) | 2단계 AI에 가장 가까움 |
| Main Coder | Opus | Core 아키텍처 + 핵심 시스템 | 기획 Workflow 범위 밖 |
| Sub Coder x2 | Sonnet | 할당 노드 구현 | 기획 Workflow 범위 밖 |
| Validator | Sonnet | 코드 5단계 검증 | 3단계 AI에 대응 가능하나, 현재는 코드 검증 전용 |
| DB Builder | Sonnet | C# 소스코드 파싱 → DB 저장 | 1단계 AI에 대응하나, 현재는 코드 파싱 전용 |

**시사점:**
- 현재 Agent 팀은 **코드 파이프라인 전용**으로 설계되어 있습니다
- 기획 Workflow의 "1단계 AI(기획 DB 가공)"을 담당할 Agent가 없습니다
- DB Builder(`db-builder.md`)는 C# 소스코드 파싱만 가능 — 기획서/밸런스 시트 파싱은 별도 Agent 필요
- Validator(`validator.md`)는 코드 5단계 검증만 수행 — 기획 통합 검증(3단계)은 검증 항목이 완전히 다름

**제안:**
기획 Workflow 구현 시 필요한 신규 Agent 정의:
```
Design DB Builder — 기획 자료 파싱 → 정규화 → Base Design DB 저장
Design Validator  — 기획 결과물 통합 검증 (교차 일관성, 유저 여정 시뮬레이션, 누락 검출)
```
또는 기존 Designer Agent의 역할을 확장하여 1단계+2단계를 겸임하는 방안도 있으나, 3-AI 분리 원칙과 충돌합니다.

### 1-2. Designer Agent의 현재 기능 vs 기획 문서의 2단계

**Designer Agent** (`designer.md` L69-169)의 현재 출력 구조:
```
Layer 1: game_design.yaml    — 게임 기획서
Layer 2: system_spec.yaml    — 시스템 명세서
Layer 3: nodes/*.yaml        — AI_기획서 (코드 생성용 YAML)
```

**기획 문서 2단계** (기획 L235-389)의 출력물:
```
2-1. 컨셉 문서 (디자인 필러, 핵심 루프, 타겟 유저)
2-2. 시스템 기획서 (규칙 정의, 데이터 테이블, UI 플로우)
2-3. 밸런스 시트 (성장 테이블, 전투 공식, 경제 시뮬레이션)
2-4. 콘텐츠 데이터 (몬스터 양산, 스테이지 구성, 퀘스트 체인)
2-5. BM 설계서 (과금 시뮬레이션, LTV 모델링)
```

**차이점:**
- Designer는 `game_design.yaml` → `system_spec.yaml` → `nodes/*.yaml` 순서의 **코드 생성 지원용** 출력에 초점
- 기획 문서의 2단계는 **기획 자체의 완결성**에 초점 (밸런스 시뮬레이션, 경제 모델링, BM 설계)
- 특히 `designer.md`에는 밸런스 시뮬레이션, 경제 시뮬레이션, LTV 모델링 기능이 없음

현재 Designer Agent의 `game_design.yaml`에는 `sub_genre: []` 필드가 있어서 (`designer.md` L76), 1차 리뷰에서 제기한 복합 장르 문제에 대한 **방안 A(주 장르 + 서브 장르)**가 이미 부분적으로 반영되어 있습니다. 다만 기획 문서에서는 이 필드를 활용하지 않고 `genre: Idle RPG`로 복합 표기하고 있어 불일치가 있습니다.

### 1-3. Validator Agent의 검증 체계 비교

**코드 Validator** (`validator.md` L29-76) — 5단계:
```
1. Syntax: using 문, 타입, 괄호, 세미콜론
2. Dependency: 참조 클래스 존재, 순환 참조
3. Contract: provides/requires 구현 확인
4. NullSafety: 컬렉션 null 체크, ?. 연산자
5. Pattern: Role 패턴, 금지 패턴, 최적화 규칙
```

**기획 3단계 통합 검증** (기획 L392-431) — 4가지 관점:
```
3-1. 교차 일관성: 시스템↔밸런스, 콘텐츠↔시스템, BM↔밸런스, UX↔시스템
3-2. 유저 여정 시뮬레이션: Day 1~30 경험 추적
3-3. 누락 검출: 참조하는 아이템/스킬/스탯이 정의되어 있는지
3-4. AI 자가 검증 주석: 검증 과정 발견 사항을 주석으로 추가
```

**핵심 차이:** 코드 검증은 **구문 + 구조 + 패턴** 기반이고, 기획 검증은 **의미 + 맥락 + 시뮬레이션** 기반입니다. 현재 Validator Agent를 그대로 확장하기는 어렵고, 기획 전용 검증 로직이 필요합니다.

**Validator의 피드백 카테고리 비교:**

코드 Validator (`validator.md` L107-116):
```
PERF, LOGIC, PATTERN, READABLE, SECURITY, CONTRACT, ROLE (7개)
```

기획 피드백 (기획 L474-489):
```
SYSTEM (3개), BALANCE (4개), CONTENT (2개), BM (2개), UX (2개), DIRECTION (1개) → 총 14개
```

코드와 기획의 피드백 카테고리가 완전히 다른 것은 당연하고 적절합니다. 기획 문서의 14개 카테고리가 기획 도메인 9종에 비해 다소 적지만, 초안으로서는 핵심 영역을 잘 커버하고 있습니다.

## 2. AI Tester 시스템과의 연계

### 2-1. AI Tester C10+ v2.5 사양서와의 접점

기획 문서는 AI Tester를 여러 곳에서 참조합니다:
- **1단계**: 외부 게임 분석 (기획 L221-227)
- **1단계**: 자사 게임 실제 구현 검증 (기획 L218)
- **7단계**: 플레이 기반 검증 (기획 L592-674)

**AI Tester C10+ v2.5 사양서** (`C10_PLUS_V25_AI_TESTER_SYSTEM.md`)의 핵심:
```
6-Phase 파이프라인:
  Phase 1: CAPTURE (10개 미션별 스크린샷 수집)
  Phase 1.5: OCR (수치 영역 자동 추출)
  Phase 2: VISION (Claude Sonnet 세션별 분석)
  Phase 2.5: WIKI (커뮤니티 데이터 교차검증)
  Phase 3: AGGREGATE (도메인 가중 합의)
  Phase 4: SPEC (32개 파라미터 YAML 추정)
```

출력 형식:
```yaml
parameters:
  - id: ANV01
    name: max_chapter
    value: 20
    confidence: high
    source: "관찰"
```

### 2-2. AI Tester 출력 → Base Design DB 변환 과정 미정의

기획 문서 L227:
```
32개 파라미터 추정 결과를 Base Design DB로 가공 (source: observed)
```

이 한 줄이 전부입니다. 실제로는 AI Tester의 32개 파라미터(YAML)를 기획 DB의 분류 체계(도메인 > 장르 > 시스템 + 다중 태그)로 변환하는 **매핑 규칙**이 필요합니다.

**AI Tester 32개 파라미터의 9개 카테고리:**
```
progression, growth, equipment, combat, gacha, economy, system, visual, architecture
```

**기획 문서의 9개 도메인:**
```
InGame, OutGame, Balance, Content, BM, LiveOps, UX, Social, Meta
```

대응 관계가 1:1이 아닙니다:

| AI Tester 카테고리 | 기획 도메인 | 비고 |
|-------------------|------------|------|
| progression | Content + Balance | 레벨 진행은 콘텐츠, 경험치 곡선은 밸런스 |
| growth | Balance | 성장 수치 |
| equipment | OutGame + Balance | 장비 시스템 + 장비 밸런스 |
| combat | InGame + Balance | 전투 시스템 + 전투 밸런스 |
| gacha | OutGame + BM | 가챠 시스템 + 과금 연관 |
| economy | Balance + BM | 경제 밸런스 + 수익 모델 |
| system | InGame | 핵심 시스템 구조 |
| visual | UX | UI/연출 |
| architecture | (해당 없음) | 코드 구조 → 기획 DB에 불필요 |

이 매핑 테이블을 기획 문서에 포함하거나, 별도 변환 사양으로 정의해야 합니다.

### 2-3. AI Tester 정확도와 기획 DB 신뢰도

`C10_Plus_개선방안.md`에 따르면:
```
v1: 89.5% (퍼즐 3종 평균)
v2.5 목표: 93-95% (OCR + Wiki + APK 보강)
관찰 천장: ~90.6% (32개 중 29개 관찰 가능)
```

기획 문서 L225:
```
AI Tester 10명 관찰만으로 약 85~89.5% 정확도 달성, Workflow 참조용으로 충분
```

이 수치가 정확히 일치합니다. 다만 기획 문서에서 "참조용으로 충분"이라고 했지만, **DB에 저장된 데이터의 10~15%가 부정확**하다면 그 데이터를 참조한 기획도 영향을 받습니다.

**제안:** `source: observed`인 데이터에 `accuracy_estimate: 0.895` 같은 메타데이터를 함께 저장하여, AI가 참조 시 "이 데이터는 약 89.5% 정확도로 추정된 관찰 데이터"임을 인지하게 하면 더 안전합니다.

### 2-4. 패턴 카드 사용 금지 근거

기획 문서 L223-226:
```
패턴 카드(Level 1/Level 2) 방식은 사용하지 않음
이유: 패턴 카드는 소스코드 역추출 경로에서 생성되었을 가능성이 있어 법적 리스크 존재
```

이 판단은 Clean Room 원칙에 부합합니다. `C10_Plus_개선방안.md`에서도 Clean Room을 "소스코드 미접근이 핵심"으로 정의하고 있고, 패턴 카드가 Base Code DB의 파싱 결과물(소스 기반)에서 파생될 수 있다는 점에서 법적 리스크 차단은 적절합니다.

다만 "AI Tester 10명 관찰만으로"라고 했을 때, v2.5에서 추가된 **APK 에셋 추출(M4)**, **커뮤니티 위키 교차검증(M3)** 기능은 관찰 이외의 보조 데이터를 사용합니다. 기획 문서에서 이를 허용하는지 여부를 명시하면 좋겠습니다.

## 3. DB 구조와의 정합성

### 3-1. 코드 DB 물리 구조 vs 기획 DB 필요 구조

**현재 코드 DB** (`db-builder.md` L100-150):
```
db/base/{genre}/{layer}/
├── index.json          # 경량 인덱스
└── files/
    └── {fileId}.json   # 상세 정보

인덱스 항목: fileId, layer, genre, role, system, score, provides, requires
상세 파일: fileId, filePath, namespace, layer, genre, role, system, score, usings, classes[]
```

**기획 DB에 필요한 구조** (기획 문서에서 추론):
```
db/design/base/{genre}/{domain}/
├── index.json
└── files/
    └── {designId}.json

인덱스 항목: designId, domain, genre, system, score, source, tags
상세 파일:
  - Layer 1 (프로젝트 통짜): 전체 기획 구조, 시스템 관계도
  - Layer 2 (세부 요소): 개별 공식, 테이블, 확률 등
```

기획 문서의 2레이어 구조(기획 L63-74)는 코드 DB의 단일 레이어보다 복잡합니다. 코드는 클래스 단위로 깔끔하게 분리되지만, 기획은 "프로젝트 전체 맥락"과 "개별 요소"를 동시에 보존해야 하기 때문입니다. 이 설계 자체는 좋으나, 물리적 구현 방식을 정의해야 합니다.

### 3-2. Rules 구조 비교

**현재 코드 Rules** (`db/rules/generic_rules.json`):
```json
{
  "ruleId": "null-check-collection",
  "type": "Generic",
  "genre": null,
  "category": "LOGIC.NULL_REF",
  "pattern": "list.First()",
  "solution": "list.FirstOrDefault()",
  "frequency": 0
}
```

**기획 Rules** (기획 L580-589):
```
| Generic 규칙  | "재화 Source 없이 Sink만 있는 구간 3일 이상 금지" |
| 장르별 규칙   | "Idle RPG 방치보상 최대 축적 8~24시간 권장" |
| 도메인별 규칙 | "가챠 천장 미설정 시 유저 이탈 위험 경고" |
```

코드 Rules에는 `pattern → solution` 구조가 있지만, 기획 Rules는 `조건 → 경고/권장` 구조가 적합합니다. 기획 Rules의 JSON 형식을 정의할 때 코드와 동일한 형식을 강제하면 자연어 기반 기획 규칙을 담기 어려울 수 있습니다.

**제안:** 기획 Rules 형식 예시:
```json
{
  "ruleId": "no-sink-without-source",
  "type": "Generic",
  "domain": "Balance",
  "genre": null,
  "category": "BALANCE.ECONOMY_IMBALANCE",
  "condition": "재화 Sink만 있고 Source가 없는 구간이 3일 이상",
  "recommendation": "해당 구간에 일일 퀘스트/보상 등 Source 추가",
  "severity": "warning",
  "frequency": 0
}
```

## 4. WORKFLOW.md와의 파이프라인 연결

### 4-1. 현재 파이프라인에서 기획 Workflow의 위치

**WORKFLOW.md L72-86** — 현재 파이프라인:
```
Phase 1: DB 구축 (DB Builder)
Phase 2: 기획서 생성 (Designer)
Phase 3: 코드 생성 (Coder x N 병렬)
Phase 4: 검증 & 축적 (Validator)
```

**기획 문서 L599-607** — 코드와의 연결:
```
기획 Workflow 3단계(통합 검증) 통과
    ↓ 기획서를 코드 Workflow 2단계(시스템 명세서)로 자동 변환
코드 Workflow 2~6단계 진행
    ↓ 플레이 가능한 빌드 생성
기획 Workflow 7단계(플레이 검증) 진행
```

이 연결 지점은 명확합니다. 기획 Workflow의 출력물이 코드 Workflow Phase 2의 입력이 되는 구조입니다. 다만 "자동 변환"이 어디까지 자동인지는 정의가 필요합니다.

현재 Designer Agent의 출력(`game_design.yaml` → `system_spec.yaml` → `nodes/*.yaml`)은 이미 코드 생성에 최적화된 형식이므로, 기획 문서 2단계의 출력물(기획서, 밸런스 시트 등)을 이 형식으로 변환하는 과정이 추가로 필요합니다.

### 4-2. 기획 7단계 → 코드 파이프라인 역방향 피드백

기획 문서 L624:
```
차이가 큰 부분 → 기획 Workflow 2단계로 되돌아가 밸런스 수정
```

이 역방향 피드백은 현재 WORKFLOW.md에 정의되어 있지 않습니다. 코드 파이프라인은 Phase 4(검증) 실패 시 해당 Coder에게만 피드백을 보내는 단방향 구조입니다(WORKFLOW.md L237-253). 기획 7단계 검증 결과가 기획 2단계로 되돌아가면, 코드도 함께 재생성해야 하는 상황이 발생할 수 있습니다.

**제안:** 전체 파이프라인 순서도에 기획↔코드 간 역방향 피드백 경로를 추가:
```
기획 7단계 플레이 검증 실패
  → 밸런스 이슈: 기획 2단계 밸런스 수정만 (코드 재생성 불필요)
  → 시스템 이슈: 기획 2단계 시스템 수정 → 코드 Phase 2부터 재시작
  → 양쪽 이슈: 기획 2단계 수정 → 코드 전체 재생성
```

## 5. 7단계 플레이 검증 실현 가능성

### 5-1. AI Tester 인프라 활용 (기획 L609-626)

기획 문서에서 "AI Tester 인프라(BlueStacks + ADB)를 그대로 활용한다"고 했는데, C10+ v2.5 사양서를 보면:

**현재 AI Tester 인프라:**
- BlueStacks 에뮬레이터 + ADB 제어
- Smart Player: Vision 기반 자율 탐색 (classifier.py + navigator.py + mission_router.py)
- 10개 미션별 전문 관찰 프롬프트

**자사 게임 가속 테스트에 필요한 추가 요소:**
- 서버 통신 제어 (오프라인 모드 또는 로컬 서버)
- 시간 가속 API (서버 측 시간 스킵)
- 30일치 데이터 수집 및 분석 파이프라인

현재 AI Tester는 **외부 게임 관찰**에 최적화되어 있고, 자사 게임의 가속 테스트는 별도 구현이 필요합니다. 특히 서버 가속 환경은 게임 서버 팀의 협조가 필요한 영역입니다.

### 5-2. AI 가상 유저 대규모 테스트 (기획 L647-674)

기획 문서 L649:
```
AI Tester 관찰 정확도가 90%+ 달성되면, AI 가상 유저 에이전트를 생성하여
천~만 명 규모 동시 시뮬레이션이 가능해집니다.
```

`C10_Plus_개선방안.md`의 정확도 로드맵:
```
v1: 89.5% → v2.5 목표: 93-95%
관찰 천장: ~90.6%
```

v2.5에서 90%+를 달성할 수 있으므로 이 기능의 전제 조건은 충족 가능합니다. 다만 "관찰 정확도 90%"와 "가상 유저 시뮬레이션"은 다른 문제입니다:

- **관찰 정확도**: 화면을 보고 파라미터를 추정하는 능력
- **가상 유저**: 추정된 파라미터 기반으로 게임을 "플레이"하는 에이전트

가상 유저는 관찰 정확도와 별개로, 유저 행동 모델(결정 트리, 과금 성향, 이탈 패턴 등)이 필요합니다. 이 부분은 별도 사양서가 필요할 수 있습니다.

## 6. 8단계 라이브 동기화 — 운영 인프라 연계

### 6-1. 라이브 데이터 수집 경로

기획 문서 L718-734:
```
1. 라이브 패치 발생
2. 변경사항 DB 반영 (새 버전 추가)
3. 현행 버전 표시
4. 시즌/대규모 패치 단위 스냅샷
```

이 과정에서 **누가/어떻게** 라이브 데이터를 DB에 반영하는지가 미정의입니다:

- **수동 입력**: 기획자가 패치 때마다 DB에 수기 등록 → 현실적이지 않음
- **자동 파싱**: 게임 서버/클라이언트 데이터 테이블 변경을 감지하여 자동 DB 반영 → 서버 연동 필요
- **반자동**: 기획자가 변경 사항을 표준 양식으로 작성 → AI가 DB 형식으로 변환

기획 초안에서는 구현 방식까지 다루지 않는 것이 맞지만, "반자동 입력 + AI 자동 변환"이 현실적이라는 방향성 정도는 제시하면 좋겠습니다.

### 6-2. KPI 보고서와의 연계

현재 코드 Validator는 프로젝트 완료 시 KPI 보고서를 자동 생성합니다:

**WORKFLOW.md L551-570:**
```
KPI 측정 항목: 디버깅 시간, 피드백 횟수, 베이스 코드 편입 비율, 데이터셋 수 등
경로: E:\AI\History\{ProjectName}\KPI.md
```

기획 문서의 라이브 버전 관리에서 KPI 데이터를 기록합니다(기획 L710-713):
```yaml
live_context:
  trigger: "Day 14 리텐션 40% → 35% 하락"
  kpi_before: { d14_retention: 0.35 }
  kpi_after: { d14_retention: 0.42 }
```

이 두 KPI는 성격이 다릅니다:
- **코드 KPI**: 개발 프로세스 효율성 (생성 시간, 피드백 횟수)
- **기획 KPI**: 게임 비즈니스 성과 (리텐션, 매출, DAU)

둘 다 필요하지만, 저장 경로와 형식을 분리해야 합니다.

## 7. 기획 Workflow가 코드 시스템에 역으로 기여할 수 있는 부분

### 7-1. source 태그 5종 → 코드 DB에도 도입 검토

기획 문서의 source 태그 분류(기획 L129-140)는 코드 DB에도 적용 가치가 있습니다:

| source 태그 | 기획 | 코드 (도입 시) |
|-------------|------|---------------|
| internal_original | 자사 기존 기획서 | 자사 기존 프로젝트 파싱 코드 |
| internal_produced | AI 생성 + 검증 기획 | AI 생성 + 검증 코드 |
| internal_live | 라이브 수정 기획 | 라이브 핫픽스 코드 |
| observed | AI Tester 관찰 자료 | (해당 없음) |
| community | 커뮤니티 데이터 | 오픈소스 참조 코드 |

현재 코드 DB에는 출처 구분이 없어서, "자사 프로젝트에서 파싱한 검증된 코드"와 "AI가 생성한 코드"의 신뢰 성격 차이를 구분하지 못합니다.

### 7-2. 사후 라벨 분리 원칙 → 코드 DB에도 적용 검토

기획 문서의 사후 라벨 분리(기획 L556-558)는 코드에도 적용할 수 있습니다. 현재 코드의 신뢰도 점수는 "재사용 성공/실패"만 반영하는데, "재사용은 성공했지만 프로젝트 자체가 실패한 경우"를 구분하지 못합니다. 시장 성과와 코드 품질을 분리하는 원칙은 코드 DB에도 유효합니다.

### 7-3. 실패 데이터 처리 원칙

기획 문서 L577-578:
```
"왜 안 됐는지 원인을 모르는 경우"에는 반면교사로 사용하지 않습니다.
원인 불명 상태에서 "이렇게 하지 마"를 학습시키면 AI가 잘못된 규칙을 학습할 수 있습니다.
```

이 원칙은 코드 DB의 `재사용 실패: -0.15`(CLAUDE.md L134)에도 적용을 검토할 만합니다. "재사용 실패"의 원인이 코드 품질 문제인지, 프로젝트 요구사항 불일치인지를 구분하지 않고 일률 감점하면 잘못된 평가가 될 수 있습니다.

## 8. 디벨롭 제안 — 기획 문서에 추가하면 좋을 항목

### 8-1. 밸런스 시뮬레이션 엔진 명세

기획 문서 2-3단계(기획 L312-347)에서 경제 시뮬레이션, 전투 시뮬레이션 등을 언급하지만, 이것이 어떤 방식으로 실행되는지는 미정의입니다:

- LLM이 수학적 시뮬레이션을 직접 수행? (정확도 이슈)
- 별도 시뮬레이션 스크립트 실행? (Python/JS)
- 스프레드시트 기반 계산?

시뮬레이션의 신뢰성이 기획 품질에 직결되므로, 실행 방식에 대한 방향성이 있으면 좋겠습니다.

### 8-2. 기획 DB 검색 API

코드 DB는 `index.json`을 통한 검색이 가능하고, DB Builder(`db-builder.md`)가 정의한 스키마가 있습니다. 기획 DB도 유사한 검색 API(또는 인덱스 구조)를 정의하면 구현 시 바로 활용할 수 있습니다.

### 8-3. 버전 간 호환성

기획 문서 L697-714의 버전 관리 구조에서, 버전 간 스키마 변경이 발생했을 때의 호환성 정책이 없습니다. 예를 들어 데미지 공식이 v1.0.0에서 `ATK × 배율 - DEF × 0.5`였다가 v2.0.0에서 `(ATK - DEF) × 배율`로 완전히 바뀌면, 두 버전의 데이터를 어떻게 비교하는지의 문제입니다.

### 판단

기획 문서 초안의 완성도가 높아서, 위 과제들은 대부분 **구현 단계**에서 순차적으로 해결 가능합니다. 특히 과제 1~3은 코드 Workflow 측에서 먼저 구조를 잡아주면 기획 Workflow가 그에 맞춰 세부를 확정할 수 있는 구조이므로, 양쪽 협의가 필요합니다.

코드 Workflow에 역으로 도입을 검토할 만한 기획 고유 설계(source 태그, 사후 라벨 분리, 실패 데이터 원칙)도 있어서, 이후 양쪽 문서의 상호 보강이 기대됩니다.
