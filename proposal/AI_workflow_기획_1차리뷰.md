# AI 기획 Workflow 초안 — 교차 검증 리뷰

> **리뷰 대상**: AI_workflow_기획.md (v1.1, 2026-02-24)
> **검증 기준**: CLAUDE.md (코드 Workflow)
> **리뷰 일자**: 2026-02-25

---

## 총평

전체적으로 코드 Workflow(CLAUDE.md)와의 구조적 대칭이 잘 설계되어 있습니다. 3-AI 분리, DB 검색 우선순위, 신뢰도 점수 체계 등 핵심 구조를 그대로 가져가면서, 기획 특성에 맞는 확장(디렉터 포지셔닝, 사후 라벨, 라이브 버전 관리)이 자연스럽게 추가되어 있습니다.

아래는 CLAUDE.md와 대조하여 **일치 확인**, **불일치/미정의**, **기획 고유 강점** 순으로 정리한 결과입니다.

## 1. 일치 확인 — 코드 Workflow와 대응이 잘 된 부분

| 항목 | CLAUDE.md (코드) | 기획 문서 | 판정 |
|------|-----------------|----------|:----:|
| AI 분리 원칙 | 3-AI (DB 가공 / 코드 생성 / 검증) | 3-AI (DB 가공 / 기획 생성 / 검증) | 일치 |
| DB 검색 우선순위 | Expert(장르) → Expert(Generic) → Base(장르) → Base(Generic) → AI 생성 | 동일 5단계 | 일치 |
| 신뢰도 초기값 | 0.4 | 0.4 | 일치 |
| Expert DB 승격 임계값 | >= 0.6 | >= 0.6 | 일치 |
| Rules 추출 | Generic 규칙 + 장르별 규칙 | Generic + 장르별 + 도메인별 규칙 | 확장 |
| 피드백 형식 | JSON (category, severity, suggestion) | JSON (category, section, direction, designPillarCheck) | 확장 |
| Generic 장르 활용 | 장르 무관 코드 → Generic 분류 | 장르 무관 기획 → Generic 분류 | 일치 |
| 검수 구조 | 중복/누락/태깅/맥락 확인 | 동일 4가지 확인 | 일치 |

기획 문서가 코드 Workflow를 **단순 복사가 아니라 기획 특성에 맞게 확장**한 점이 좋습니다:
- 피드백에 `designPillarCheck`(디자인 필러 위반 여부) 추가
- Rules에 도메인별 규칙 추가
- 검수 항목에 "맥락 보존"(세부 분리 시 원래 의도 훼손 여부) 포함

### 1-1. 3-AI 분리 원칙

**CLAUDE.md L8-11:**
```
1. **3-AI 분리** - DB 가공 AI, 기획 AI, 코드 생성 AI
2. **환각 방지** - 정규화된 분류 체계 + DB 참조 필수
3. **병렬 작업** - Agent Teams로 Phase 내 노드 병렬 생성
```

**기획 문서 L18-28:**
```
1단계 AI: Base Design DB 가공 (기획 자료 → 정규화된 DB)
2단계 AI: 기획서/밸런스/콘텐츠 생성 (디렉션 + DB 참조 → 기획 결과물)
3단계 AI: 기획 검증 및 시뮬레이션 (결과물 → 일관성/밸런스/경제 검증)
```

코드 Workflow의 "DB 가공 / 코드 생성 / 검증" 3분할을 "DB 가공 / 기획 생성 / 기획 검증"으로 정확히 대응시켰습니다. 환각 방지를 위해 역할을 분리한다는 핵심 원칙(L20: "역할 전환에 의한 환각현상 방지")도 동일합니다.

### 1-2. DB 검색 우선순위 (5단계)

**CLAUDE.md L115-124:**
```
| 1 | Expert DB (해당 장르) | genre 일치 AND score >= 0.6 |
| 2 | Expert DB (Generic)   | genre = Generic AND score >= 0.6 |
| 3 | Genre Base DB         | genre 일치 |
| 4 | Generic Base DB       | genre = Generic |
| 5 | AI_기획서 기반 생성    | 참조 코드 없음 |
```

**기획 문서 L150-156:**
```
1       Expert DB (해당 장르)           genre 일치 AND score >= 0.6
2       Expert DB (Generic)             genre = Generic AND score >= 0.6
3       Genre Base DB                   genre 일치
4       Generic Base DB                 genre = Generic
5       AI 자체 생성                    참조 자료 없음 → 디렉터 상세 디렉션 필수
```

5단계 순위가 완전 동일합니다. 기획 문서가 5순위에 "디렉터 상세 디렉션 필수"를 추가한 것은 좋은 확장입니다. 코드의 경우 AI_기획서(YAML)가 있어 참조 없이도 생성이 가능하지만, 기획의 경우 참조 데이터 없이 생성하면 품질 보장이 어렵기 때문입니다.

### 1-3. 신뢰도 점수 기본 구조

**CLAUDE.md L128-136:**
```
| 초기 저장               | 0.4    |
| 피드백 반영 완료        | +0.2   |
| 재사용 성공 (1회당)     | +0.1   |
| 재사용 실패 (1회당)     | -0.15  |
| 다른 장르에서 재사용 성공 | +0.1 (Generic 승격 검토) |
| Expert DB 승격 임계값    | >= 0.6 |
```

**기획 문서 L541-547:**
```
| 초기 저장 (3단계 통합검증 통과)       | 0.4  |
| 디렉터 검증 통과 (피드백 없이 승인)   | +0.2 |
| 피드백 반영 완료 후 승인              | +0.1 |
| 다른 프로젝트에서 구조 참조 성공      | +0.1 |
| 다른 장르에서 참조 성공              | +0.1 (Generic 승격 검토) |
```

초기값 0.4, 승격 임계값 0.6은 동일합니다. 다만 이벤트 세분화와 감점 부분에서 차이가 있습니다 (→ 2-2절에서 상세 분석).

### 1-4. 피드백 형식 확장

CLAUDE.md에는 피드백 상세 형식이 명시되어 있지 않습니다. 신뢰도 점수 체계(CLAUDE.md L128-136)에서 "피드백 반영 완료 +0.2"라는 이벤트만 있을 뿐, 피드백 자체의 카테고리나 JSON 구조는 정의되어 있지 않습니다.

**기획 문서 L457-489:**
```
targetId, domain, genre, category, section, description, direction, designPillarCheck
+ 14개 피드백 카테고리 (SYSTEM.RULE_CONFLICT, BALANCE.CURVE_TOO_STEEP 등)
```

기획 문서가 CLAUDE.md보다 피드백 체계를 훨씬 상세하게 정의했습니다. 특히 `designPillarCheck`(디자인 필러 위반 여부), `direction`(수정 방향) 필드는 기획 특성에 맞는 유의미한 확장입니다.

### 1-5. Generic 장르 활용

**CLAUDE.md L56, L80:**
```
│   ├── generic\    # 장르 무관 공통   (DB 위치)
Generic, RPG, Idle, Merge, ...                (장르 목록에 Generic 포함)
```

**기획 문서 L107-108:**
```
Generic 장르를 두어 장르 무관한 기획 패턴은 공통으로 관리하고,
검색 시 "해당 장르 → Generic 순서"로 참조하여 데이터 부족 문제를 방지합니다.
```

양쪽 모두 Generic을 첫 번째 장르로 배치하고, 장르 무관 데이터를 공통 관리하는 전략이 일치합니다.

### 1-6. 파이프라인 단계 대응

**CLAUDE.md L157-169:**
```
Phase 1: DB 구축 (1회)
Phase 2: 기획서 생성 (프로젝트마다)
Phase 3: 코드 생성 (프로젝트마다)
Phase 4: 지식 축적 (자동)
```

**기획 문서 L837-846:**
```
| 1단계: Base Code DB 가공   | 1단계: Base Design DB 가공   |
| 2단계: 기획서 제작/가공     | 2단계: 기획 생성             |
| 3단계: 코드 생성           | → 코드 Workflow로 연결       |
| 4단계: 피드백/재생성        | 4단계: 디렉터 검수/피드백     |
| 5단계: 재생성 평가          | 5단계: 재생성 평가           |
| 6단계: DB 축적             | 6단계: DB 축적              |
| -                          | 7단계: 플레이 검증           |
| -                          | 8단계: 라이브 DB 동기화       |
```

코드 4단계를 기획 8단계로 확장하면서도, 1~6단계까지의 대응 관계가 명확합니다. 코드의 Phase 3(코드 생성)에 해당하는 부분을 "코드 Workflow로 연결"로 처리한 것은 두 Workflow 간 접점을 잘 설계한 것입니다.

## 2. 불일치 / 미정의 — 확인이 필요한 항목

### 2-1. Genre 목록 불일치

| CLAUDE.md | 기획 문서 |
|-----------|----------|
| Generic, RPG, Idle, Merge, SLG, Tycoon, Simulation, Puzzle, **Playable** | Generic, RPG, Idle, SLG, Simulation, Tycoon, Merge, Puzzle, **Casual** |

**CLAUDE.md L80:**
```
Generic, RPG, Idle, Merge, SLG, Tycoon, Simulation, Puzzle, Playable
```

**기획 문서 L104:**
```
Generic(공통) / RPG / Idle / SLG / Simulation / Tycoon / Merge / Puzzle / Casual
```

CLAUDE.md에서는 `Playable`을 장르로 분류하고 있고(DB 위치: `db/base/playable/`, CLAUDE.md L63), 기획 문서에서는 `Casual`로 대체했습니다.

기획 문서의 판단("Playable은 장르가 아니라 플랫폼")에 동의합니다. CLAUDE.md 자체도 `platform: playable`로 플랫폼 분기를 별도로 두고 있어서(CLAUDE.md L176-178), 장르 목록에서 `Playable`을 빼고 `Casual`로 대체하는 게 논리적으로 맞습니다.

다만 **양쪽을 동시에 변경하지 않으면 DB 검색 키가 불일치**합니다. 현재 코드 DB는 `db/base/playable/` 디렉토리(CLAUDE.md L63)를 사용하고 있어서, 기획에서 `Casual` 장르로 검색하면 코드 DB에서는 해당 장르가 존재하지 않게 됩니다.

> **확인 필요**: `Casual`로 통일할지 결정 + 코드 쪽도 함께 수정

### 2-2. 신뢰도 점수 변동 비대칭

| 이벤트 | CLAUDE.md (코드) | 기획 문서 |
|--------|:---------------:|:--------:|
| 초기 저장 | 0.4 | 0.4 |
| 피드백 반영 완료 | +0.2 | — |
| 디렉터 검증 통과 (피드백 없이) | — | +0.2 |
| 피드백 반영 후 승인 | — | +0.1 |
| 재사용 성공 (1회당) | +0.1 | — |
| 다른 프로젝트에서 참조 성공 | — | +0.1 |
| 다른 장르에서 재사용 성공 | +0.1 (Generic 승격 검토) | +0.1 (Generic 승격 검토) |
| **재사용 실패 (1회당)** | **-0.15** | **없음** |

**CLAUDE.md L131-135:**
```
| 피드백 반영 완료         | +0.2 |
| 재사용 성공 (1회당)      | +0.1 |
| 재사용 실패 (1회당)      | -0.15 |
```

**기획 문서 L544-547:**
```
| 디렉터 검증 통과 (피드백 없이 승인)   | +0.2 |
| 피드백 반영 완료 후 승인              | +0.1 |
| 다른 프로젝트에서 구조 참조 성공      | +0.1 |
```

**a) 이벤트명 변경**: 코드의 "피드백 반영 완료 +0.2"를 기획에서는 "디렉터 검증 통과 +0.2 / 피드백 반영 후 승인 +0.1"로 분리했습니다. 사람이 최종 승인하는 기획 특성을 반영한 것이라 합리적입니다.

- **코드**: `피드백 반영 완료 = +0.2`로 단일 이벤트 (CLAUDE.md L132)
- **기획**: 디렉터(사람)가 판정 → `피드백 없이 승인 = +0.2`, `피드백 후 승인 = +0.1`로 2단계 분리 (기획 L544-545)

피드백 없이 바로 통과한 기획이 더 높은 점수를 받는 설계는 합리적입니다.

**b) 감점 메커니즘 부재**: 코드에는 `재사용 실패: -0.15`(CLAUDE.md L134)가 있는데 기획에는 없습니다. 감점이 없으면 Expert DB에 한번 올라간 기획이 영구 유지됩니다.

- 의도적이라면: "기획은 코드와 달리 재사용 실패/성공 판단이 모호하다"는 근거를 명시
- 누락이라면: "다른 프로젝트에서 참조했으나 부적합 판정: -0.15" 등 추가 검토

**c) 재사용 용어 차이**: 코드는 "재사용 성공"(CLAUDE.md L133), 기획은 "다른 프로젝트에서 구조 참조 성공"(기획 L546). 기획의 표현이 더 구체적이고 적절합니다. 코드는 클래스 단위로 재사용이 명확하지만, 기획은 "구조 참조"라는 다른 수준의 재사용이기 때문입니다.

> **확인 필요**: 감점 없는 것이 의도적 설계인지

### 2-3. 자동 점수 — CLAUDE.md에 없는 새로운 개념

기획 문서 6단계에 **자동 점수** 3종이 추가되어 있습니다.

**기획 문서 L531-537:**
```
| 논리 완결성   | 시스템 간 모순, 누락, 참조 오류 없을수록 높음 | 0~1.0 |
| 밸런스 안정성  | 경제/전투 시뮬레이션에서 이상치 없을수록 높음 | 0~1.0 |
| 구현 복잡도   | 기존 Base Code DB와 매칭률 (구현 가능성)    | 0~1.0 |
```

CLAUDE.md에는 이런 개념이 없고 이벤트 기반 신뢰도 점수만 존재합니다(CLAUDE.md L128-136). 기획 특성상 필요한 추가인데, **두 점수의 관계가 정의되지 않았습니다**:

- 자동 점수 3개가 신뢰도 초기값(0.4)을 결정하는 건지?
- 별도 독립 지표로 관리되는 건지?
- Expert DB 승격(>= 0.6)에 자동 점수가 관여하는 건지?

이 관계가 빠지면 구현 시 "점수가 두 종류인데 어떤 걸로 승격 판단하지?"가 됩니다.

한편, 자동 점수 중 "구현 복잡도"(기획 L537)는 **Base Code DB와의 매칭률**로 정의하고 있는데, 이는 기획 문서가 코드 DB까지 참조한다는 의미입니다. CLAUDE.md의 DB 위치(CLAUDE.md L52-66)에 Design DB는 정의되어 있지 않으므로, 기획 DB와 코드 DB 간 연결 구조도 함께 정의해야 합니다.

> **확인 필요**: 자동 점수와 신뢰도 점수의 관계 명확화

### 2-4. 기획 도메인 → 코드 Domain 매핑에서 LiveOps 누락

기획 문서에서 기획 도메인 9종을 정의하고(기획 L83-98), 코드 Domain으로의 매핑 테이블을 제공합니다(기획 L865-876). 그런데 9종 중 **LiveOps만** 매핑이 빠져 있습니다:

**기획 문서 L92:**
```
| LiveOps | 라이브 운영 기획 | 업데이트 주기, 이벤트 캘린더, 공지, 보상 정책 |
```

**기획 문서 L865-876 매핑 테이블:**
```
| InGame  | Battle, Skill, Character, Stage |
| OutGame | Inventory, Shop, Item |
| Balance | 각 Domain의 Calculator/Processor |
| Content | Quest, Stage, Reward |
| BM      | Shop, IAP |
| UX      | UI, Audio |
| Social  | Network, Guild |
| Meta    | Achievement, Collection |
(LiveOps → 없음)
```

CLAUDE.md의 Role 21종(CLAUDE.md L82-105)에 LiveOps에 직접 대응하는 패턴이 없어서 매핑이 어려울 수 있지만, "해당 없음" 또는 "추후 정의"라도 명시하면 의도적인지 구분됩니다.

또한 LiveOps는 기획 문서 8단계(L678-768)에서 핵심적으로 다루는 도메인이므로, 매핑이 빠진 것이 의도적이라면 "LiveOps는 코드 Domain이 아니라 운영 프로세스에 매핑된다" 등의 설명이 필요합니다.

> **확인 필요**: LiveOps 매핑 추가 또는 "해당 없음" 명시

Generic(공통) / RPG / Idle / SLG / Simulation / Tycoon / Merge / Puzzle / Casual

**기획 문서 L118** — 바로 아래 태깅 예시:

CLAUDE.md L80에도 복합 장르 처리 규칙은 없으므로(단일 장르 9종만 나열), 양쪽 공통으로 규칙을 정해야 합니다:
- **방안 A**: 주 장르 1개 + 서브 장르 배열 (`genre: Idle`, `sub_genre: [RPG]`)
- **방안 B**: 복합 장르를 분류 체계에 추가 (`Idle RPG`를 독립 장르로)
- **방안 C**: DB 검색 시 주 장르로만 검색하고, 서브 장르는 참고 태그로만 활용

참고로 CLAUDE.md에는 복합 장르에 대한 언급이 전혀 없어서, 이 문제는 양쪽 문서 공통으로 해결해야 합니다.

### 2-6. 분류 체계 구조 차이

두 문서의 분류 체계가 상이한 축을 사용합니다:

**CLAUDE.md L70 — 코드 분류:**
```
Layer (3종: Core/Domain/Game) > Genre (9종) > Role (21종) > Tag (대기능 7 + 소기능 11)
```

**기획 문서 L78-81 — 기획 분류:**
```
기획 도메인 (9종) > Genre (9종) > 시스템 단위
+ 다중 태그 (밸런스 영역, 데이터 유형, 성과 등급, 출처, 버전)
```

기획 문서 L852-857에서 이 대응 관계를 표로 정리하고 있습니다:
```
| Layer (Core/Domain/Role)  | 기획 도메인 (InGame/OutGame/Balance/...) |
| Genre                     | Genre                                    |
| 대기능                    | 시스템 단위                               |
| 소기능 Tag                | 다중 태그                                |
```

여기서 코드의 "Layer"를 기획의 "기획 도메인"에 대응시키고 있는데, 두 분류의 성격이 다릅니다:

- **코드 Layer**: 의존성 계층 (Core → Domain → Game). 코드의 빌드 순서를 결정하는 구조적 분류.
- **기획 도메인**: 업무 분업 영역 (InGame, OutGame, Balance 등). 기획자의 역할 분담을 반영한 분류.

이 차이가 의도적인 것은 이해되지만, 매핑 테이블(기획 L852-857)에서 "가장 큰 분류 단위"로 동일시하고 있어서 혼란의 여지가 있습니다. 코드의 Layer 3종과 기획의 도메인 9종은 대응 관계라기보다는 **각 영역에 최적화된 별도 분류**라는 점을 명시하면 좋겠습니다.

### 2-7. 기획 DB 물리적 저장 구조 미정의

**CLAUDE.md L52-66** — 코드 DB의 물리적 구조가 명확합니다:
```
E:\AI\db\
├── base\           # Base Code DB (파싱된 소스)
│   ├── generic\
│   ├── rpg\
│   └── ...
├── expert\         # 검증된 코드 (score >= 0.6)
└── rules\          # 축적된 피드백 규칙
```

기획 문서에서는 **논리적 구조**(레이어 1 = 프로젝트 단위, 레이어 2 = 세부 요소)는 잘 정의했으나(기획 L63-74), **물리적 저장 위치와 파일 형식**이 정의되어 있지 않습니다:

- Base Design DB는 어디에 저장되는가? (`db/base_design/`? 코드 DB와 같은 `db/` 안?)
- Expert Design DB는? (`db/expert_design/`?)
- 인덱스와 상세 파일의 구조는? (코드처럼 `index.json + files/{id}.json`?)
- 레이어 1(프로젝트 통짜)과 레이어 2(세부 태깅)의 물리적 분리는?

이 부분은 구현 시 반드시 정의해야 하지만, 초안 단계에서는 논리적 설계가 우선이므로 긴급도는 낮습니다.

## 3. 기획 문서 고유 강점 — 코드 Workflow에 없는 가치

단순히 코드 구조를 기획에 옮긴 게 아니라, 기획 고유의 확장이 잘 되어 있는 부분:

| 항목 | 내용 | 평가 |
|------|------|------|
| **디렉터 포지셔닝** | DB 축적에 따라 디렉션 상세도가 3단계로 줄어드는 비전 (L46~49) | 실무적으로 설득력 있고, 도입 초기 저항을 줄이는 좋은 설계 |
| **입력 자료 현실적 처리** | "밸런스 시트만 있는 경우", "기획서와 구현이 다른 경우" 등 5가지 상태별 대응 (L181~189) | 과거 자료가 깔끔하지 않은 현실을 정확히 반영 |
| **실패 데이터 처리** | 원인 불명이면 반면교사로 쓰지 않는다 (L577~578) | AI 학습 오염 방지에 핵심적인 원칙 |
| **사후 라벨 분리** | 시장 성과(S/A/B/C/F)를 신뢰도 점수와 혼합하지 않고 별도 필드로 관리 (L556~558) | 마케팅/타이밍 등 외부 변수 오염 방지. 정확한 판단 |
| **source 태그 세분화** | internal_original / internal_produced / internal_live / observed / community 5종 (L129~140) | 코드에는 없는 출처 구분. "실증 데이터"와 "AI 생성 데이터"의 신뢰 성격 차이를 반영 |
| **2레이어 DB 구조** | 프로젝트 단위 통짜(맥락 보존) + 세부 요소 태깅(검색/비교) (L63~74) | 코드 DB의 단일 레이어보다 기획 자료 특성에 맞는 설계 |
| **라이브 DB 버전 관리** | 변경 이유(trigger) + KPI 전후 비교를 함께 기록 (L688~714) | 차기 프로젝트에서 "비슷한 상황에 어떻게 대응했는지" 인과관계까지 참조 가능 |
| **7단계 플레이 검증** | 문서 레벨 검증을 넘어 실제 플레이 기반 실증 (L592~674) | 코드에 없는 단계. 기획 품질의 최종 관문 |
| **8단계 라이브 동기화** | 출시 후 데이터가 DB로 순환하는 구조 (L678~768) | 장기 DB 축적 관점에서 파이프라인 완결성을 높임 |

### 3-1. 디렉터 포지셔닝 (L30-49)

기획 문서가 "기획자 = AI 도구 사용자"가 아니라 "기획자 = 디렉터"로 포지셔닝한 점은 전략적으로 좋습니다:

```
DB 부족: "데미지 공식은 ATK×배율-DEF×0.5로, 최소 데미지는 ATK의 5%로 해줘"
DB 보통: "데미지 공식은 감산식으로, 레퍼런스는 ㅇㅇ게임 기반인데 방어 비중을 좀 낮춰줘"
DB 풍부: "RPG 전투 밸런스 잡아줘, 초반은 레퍼런스보다 쉽게"
```

이 3단계 비전은 코드 Workflow에는 없는 기획 고유 설계입니다. 코드는 AI_기획서(YAML)가 있으면 DB 양과 무관하게 생성 가능하지만, 기획은 DB가 쌓일수록 디렉터의 부담이 줄어드는 점진적 개선 경로를 제시합니다.

### 3-2. source 태그 5종 분리 (L127-140)

이 설계는 코드 Workflow에도 역으로 도입을 검토할 만합니다:

```
internal_original  → 자사 기존 프로젝트에서 입력한 완성된 자료
internal_produced  → Workflow 과정을 통해 생산되어 디렉터 검증을 거친 자료
internal_live      → 라이브 운영 중 수정/업데이트된 자료
observed           → AI Tester 관찰로 추출한 외부 게임 분석 자료
community          → 위키, 가이드 등 공개 커뮤니티 데이터
```

코드도 "기존 프로젝트에서 파싱한 코드(internal_original)"와 "AI가 생성하고 검증된 코드(internal_produced)"의 신뢰 성격이 다르기 때문입니다.

### 3-3. 사후 라벨 분리 (L549-558)

```
사후 라벨은 신뢰도 점수에 직접 반영하지 않고 별도 필드로 관리합니다.
이유: 시장 성과는 기획 외 요인(마케팅, 출시 타이밍 등)에 영향받으므로,
기획 자체의 품질 점수와 혼합하면 DB 오염이 발생합니다.
```

이 판단은 정확합니다. 시장 성과를 기획 품질과 분리한 것은 DB 정합성을 위해 필수적인 설계입니다.

### 확인 필요 항목

| # | 항목 | 긴급도 | 비고 |
|:-:|------|:------:|------|
| 1 | Genre 통일 (Playable vs Casual) | 높음 | 양쪽 DB 검색 키에 영향 |
| 2 | 자동 점수 / 신뢰도 점수 관계 | 높음 | Expert DB 승격 기준 불명확 |
| 3 | 복합 장르 처리 규칙 | 중간 | 문서 내부 불일치 해소 필요 |
| 4 | 감점 메커니즘 의도 확인 | 중간 | 코드와 비대칭 |
| 5 | LiveOps 매핑 | 낮음 | 매핑 테이블 완결성 |
| 6 | 분류 체계 대응 관계 명확화 | 낮음 | Layer vs 도메인 성격 차이 설명 |
| 7 | 기획 DB 물리적 저장 구조 | 낮음 | 구현 착수 시 정의 필요 |

### 판단

1~2번은 구현 착수 전 확정이 필요하고, 3~5번은 구현 초기에 잡아도 되는 수준입니다. 6~7번은 다음 버전에서 보완해도 됩니다. 전체적으로 코드 Workflow와의 정합성이 높아서, 위 항목들만 정리하면 바로 다음 단계로 넘어갈 수 있다고 봅니다.
